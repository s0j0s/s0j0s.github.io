Claude Code Engineering Roadmap

Week 3

CLAUDE.md와 Context Engineering

반복해서 설명하는 팀 규칙은 매번 프롬프트로 붙이지 말고, Claude가 읽을 수 있는 프로젝트 메모리로 관리해야 합니다.

이번 주 핵심

CLAUDE.md는 문서라기보다 작업 지시서에 가깝습니다. 팀의 암묵지를 Claude가 판단에 쓸 수 있는 형태로 바꾸는 파일입니다.

팀에서 Claude Code를 쓰다 보면 같은 말을 반복하게 됩니다. “테스트는 이 명령으로 돌려”, “이 폴더는 건드리지 마”, “API 응답 타입은 여기서 관리해”, “리뷰할 때는 회귀 위험을 먼저 봐.” 이런 설명을 매번 요청에 붙이면 피곤하고 빠뜨리기도 쉽습니다.

Context Engineering은 Claude가 판단할 때 필요한 배경을 정리하는 일입니다. 그 중심에 CLAUDE.md가 있습니다. 좋은 CLAUDE.md는 Claude에게 프로젝트의 사용 설명서를 주는 것이 아니라, 작업할 때 지켜야 할 기준을 줍니다.

1. 왜 필요한가

Claude는 현재 대화에서 주어진 정보와 직접 읽은 파일을 바탕으로 판단합니다. 프로젝트의 관습, 금지된 패턴, 팀이 중요하게 보는 리뷰 기준은 코드만 읽어서는 알기 어렵습니다. 그래서 반복되는 설명을 프로젝트 메모리로 올려야 합니다.

효과는 단순합니다. 요청은 짧아지고, 결과는 일관됩니다. 새로 온 팀원도 CLAUDE.md를 통해 팀이 Claude Code를 어떻게 쓰는지 빠르게 이해할 수 있습니다.

2. 들어가면 좋은 내용

모든 정보를 넣을 필요는 없습니다. Claude가 실제 작업 중 자주 판단해야 하는 내용을 우선합니다.

  • 프로젝트 구조와 주요 디렉터리 역할
  • 빌드, 테스트, 실행 명령
  • 코드 스타일과 네이밍 규칙
  • 아키텍처 원칙과 의존성 방향
  • 건드리면 안 되는 파일이나 금지 패턴
  • 도메인 용어와 제품 맥락
  • Claude가 자주 실수하는 부분
  • 리뷰할 때 우선순위가 높은 기준

3. 좋은 CLAUDE.md의 느낌

좋은 파일은 길고 친절한 소개 문서가 아닙니다. Claude가 작업 중 바로 적용할 수 있는 짧은 기준의 모음에 가깝습니다.

덜 좋은 방식

우리 서비스는 고객에게 좋은 경험을 제공하는
중요한 플랫폼입니다. 코드는 깨끗하게 작성하고
가능하면 테스트를 추가합니다.

더 나은 방식

- API 타입은 src/api/types.ts에서만 정의한다.
- 결제 로직 수정 후에는 npm run test:billing을 실행한다.
- UI 문구 변경은 ko.json과 en.json을 함께 수정한다.
- 인증 흐름은 직접 리팩토링하지 말고 먼저 계획을 제안한다.

4. 너무 긴 메모리의 문제

CLAUDE.md가 길어질수록 모든 내용이 같은 중요도로 반영되기 어렵습니다. 오래된 규칙, 예외가 많은 설명, 특정 사건의 회고가 계속 쌓이면 Claude가 무엇을 우선해야 할지 흐려질 수 있습니다.

그래서 팀 메모리는 주기적으로 정리해야 합니다. 지금도 유효한 규칙인지, 실제 작업에 쓰이는지, 더 짧게 표현할 수 있는지 확인합니다. 길이를 줄이는 것이 목적이 아니라 판단 기준을 선명하게 만드는 것이 목적입니다.

5. 개인 메모리와 팀 메모리

개인 취향과 팀 규칙은 섞지 않는 편이 좋습니다. 개인 메모리에는 선호하는 응답 방식이나 개인 워크플로를 둡니다. 팀 메모리에는 모든 팀원이 공유해야 하는 빌드, 테스트, 설계, 리뷰 기준을 둡니다.

팀에서 합의하면 좋은 기준

CLAUDE.md에는 “알아두면 좋은 정보”보다 “작업할 때 반드시 반영해야 하는 기준”을 먼저 적습니다.
