Claude Code Engineering Roadmap
Week 1 of 8 기본 습관

Week 1

Claude Code에게 일을 잘 맡기는 법

“고쳐줘”라고 말하는 것에서 한 걸음만 더 나아가면, Claude Code를 작업 흐름 안에서 조금 더 자연스럽게 활용할 수 있습니다.

3줄 요약

  • AI 코딩은 질문을 잘 쓰는 것뿐 아니라 작업 흐름을 어떻게 잡을지도 중요해지고 있습니다.
  • Claude Code가 먼저 조사하고 변경 근거를 남기게 하면 수정 결과를 검토하기 쉬워집니다.
  • 이번 글에서는 “바로 구현하지 말고 먼저 조사해줘”라는 요청 방식이 어떤 의미인지 살펴봅니다.

Claude Code를 처음 쓰면 자연스럽게 이렇게 말하게 됩니다. “이 코드 고쳐줘”, “이 에러 해결해줘”, “테스트 만들어줘.” 틀린 사용법은 아닙니다. 작은 작업에서는 이 정도 요청만으로도 꽤 도움이 됩니다.

실제 회사 코드에서는 버그 하나를 고칠 때도 프로젝트 구조, 기존 설계, 테스트 방법, 배포 영향과 팀 규칙을 함께 봐야 합니다. 우리 팀에는 이런 기준을 담은 CLAUDE.md가 있으므로, 이번 글에서는 그 메모를 바탕으로 리뷰하기 쉬운 작업 요청을 만드는 방법을 살펴보겠습니다.

AI 모델이 원하는 결과를 내도록 지시, 맥락, 예시, 출력 형식을 설계하고 다듬는 과정을 Prompt Engineering이라고 부릅니다. 여기에 Claude Code처럼 파일을 읽고, 코드를 수정하고, 명령을 실행하는 도구를 사용하면 작업 순서와 검증 방법도 함께 생각해볼 수 있습니다.

이처럼 AI 에이전트가 구현을 수행할 수 있도록 맥락과 작업 범위를 정하고, 사람이 계획, 검토, 검증을 책임지는 접근을 최근에는 Agentic Engineering이라고 부르기도 합니다. 이번 뉴스레터에서는 이 개념을 거창한 이론으로 설명하기보다, 실제 업무에서 사용할 요청 문장과 체크 포인트로 풀어보겠습니다.

1. 흔한 시작: 바로 고쳐달라고 하기

가장 쉬운 요청은 바로 수정 요청입니다. 

상태 업데이트 버그 고쳐줘.
이 테스트 실패 고쳐줘.
이 서비스 코드 리팩토링해줘.

이 방식은 빠릅니다. 하지만 Claude Code가 원인을 충분히 확인하지 않은 채 눈에 띄는 위치부터 수정할 수 있습니다. 그러면 처음에는 편해 보여도 리뷰할 때 변경 이유와 테스트 근거를 다시 확인하게 됩니다.

짧게 요청한 예시

로그인 버그 고쳐줘.

조사부터 요청한 예시

로그인 실패 원인을 먼저 조사해줘.
관련 파일과 재현 흐름을 정리하고,
가장 작은 수정안을 제안한 뒤,
테스트 방법과 기존 기능에 영향을 줄 수 있는 부분까지 알려줘.

여기서 영향 범위란 이번 수정과 함께 확인해야 하는 주변 동작을 뜻합니다. 예를 들어, 로그인 로직을 고쳤다면 정상 로그인뿐 아니라 비밀번호 오류, 계정 잠금, 세션 생성, 로그인 이력 저장 같은 흐름도 같이 볼 수 있습니다.

2. 요청은 길이보다 순서가 중요할 때가 있습니다

Claude Code에게 작업을 요청할 때는 프롬프트를 길게 쓰는 것보다, 어떤 순서로 확인하게 할지를 정하는 것이 더 도움이 될 때가 있습니다. 바로 구현시키기 전에 먼저 봐야 할 것, 바꿔도 되는 범위, 확인하면 좋을 포인트를 알려주면 결과를 함께 검토하기 쉬워집니다. 

목표 정의
-> 원인 조사
-> 작은 수정안
-> 구현
-> 테스트
-> 변경 영향 검토

이런 순서로 요청하면 Claude Code의 작업 결과에 함께 남는 정보가 달라집니다. 코드만 나오는 것이 아니라, 변경 이유와 검증 근거가 같이 남기 때문입니다.

출력 형식을 정해두면 리뷰하기 편합니다

작업 결과를 매번 비슷한 형식으로 받으면 리뷰하기 쉽습니다. 아래처럼 결과 형식까지 같이 요청할 수 있습니다. 

결과는 아래 형식으로 정리해줘.
1. 원인 후보
2. 관련 파일
3. 현재 동작 / 재현 흐름
4. 수정 방향
5. 테스트 방법
6. 기존 기능에 영향을 줄 수 있는 부분

3. 리뷰하기 쉬운 결과를 받으려면

Claude Code를 사용할 때는 단순히 코드를 빠르게 만드는 것보다, 변경 이유와 확인 근거를 남기는 관점도 같이 볼 수 있습니다. 이런 정보가 함께 남으면 리뷰 부담이 줄고, 다른 팀원이 이어서 이해하기도 쉬워집니다.

  • 관련 파일을 먼저 찾아보게 합니다.
  • 수정 전에 원인과 계획을 짧게 물어봅니다.
  • 가능하면 작은 변경부터 맡겨봅니다.
  • 테스트, 빌드, 재현 조건처럼 확인 가능한 근거를 같이 남겨봅니다.
  • 마지막에는 수정한 코드가 다른 기능에 영향을 주지 않는지 한 번 더 봅니다.

이런 방식이 쌓이면 Claude Code를 개인 생산성 도구뿐 아니라 다음 사람이 이어보기 쉬운 작업 메모를 남기는 도구로도 활용할 수 있습니다.

4. 참고해볼 요청 예시

작업 요청을 조금 더 정리해서 맡기고 싶을 때는 아래와 같은 문장을 참고해볼 수 있습니다.

요청 예시

아래는 상황에 맞게 줄이거나 바꿔서 사용할 수 있는 예시입니다.

이 작업은 바로 구현하지 말고 먼저 조사해줘.
관련 파일과 현재 동작을 정리하고,
수정 방향을 2가지 이하로 비교한 뒤,
테스트로 확인할 수 있는 방법까지 제안해줘.

핵심은 속도를 늦추자는 것이 아닙니다. 리뷰할 근거를 구현 전에 먼저 만들어두자는 것입니다.

한 문장으로 정리

Claude Code를 사용할 때는 “빨리 만들어줘”뿐 아니라 “먼저 보고, 작게 고치고, 확인해서 알려줘”라는 방식도 함께 생각해볼 수 있습니다.
