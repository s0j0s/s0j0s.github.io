Claude Code Engineering Roadmap
Week 1 of 8 기본 습관

Week 1

Claude Code에게 일을 잘 맡기는 법

“고쳐줘”라고 말하는 것에서 한 걸음만 더 나아가도, Claude Code를 우리 팀 작업 흐름에 더 자연스럽게 녹일 수 있습니다.

3줄 요약

  • AI 코딩은 질문을 잘 쓰는 것뿐 아니라 작업 흐름을 어떻게 잡을지도 중요해지고 있습니다.
  • Claude Code는 바로 고치게 하기보다 먼저 조사하고, 작게 고치고, 근거를 남길 때 더 안정적으로 쓸 수 있습니다.
  • 이번 글에서는 “바로 구현하지 말고 먼저 조사해줘”라는 요청 방식이 어떤 의미인지 살펴봅니다.

이번 주 포인트

Claude Code를 단순히 코드 작성 도구로만 보면 놓치는 부분이 있습니다. 실제 업무 코드에서는 관련 파일, 현재 동작, 수정 이유, 테스트 방법을 같이 보는 방식도 참고해볼 만합니다.

Claude Code를 처음 쓰면 자연스럽게 이렇게 말하게 됩니다. “이 코드 고쳐줘”, “이 에러 해결해줘”, “테스트 만들어줘.” 틀린 사용법은 아닙니다. 작은 작업에서는 이 정도 요청만으로도 꽤 도움이 됩니다.

문제는 실제 회사 코드가 그렇게 단순하지 않다는 데 있습니다. 버그 하나를 고치려 해도 프로젝트 구조, 기존 설계, 테스트 방법, 배포 영향, 팀의 코딩 규칙을 같이 봐야 합니다. Claude Code는 파일을 읽고, 코드를 수정하고, 명령을 실행할 수 있기 때문에 이런 작업 흐름에 잘 맞습니다.

우리 팀에는 이미 Claude Code가 참고할 공통 메모인 CLAUDE.md가 있습니다. 그래서 1주차에서는 그 파일을 새로 만드는 법보다, 그 메모를 바탕으로 Claude Code에게 작업을 요청할 때 어떤 정보를 먼저 정리하게 하면 리뷰하기 쉬운지 살펴보겠습니다. 각자 잘 쓰는 것도 중요하지만, 팀에서 비슷한 방식으로 맡기면 리뷰와 인수인계가 훨씬 쉬워집니다.

좋은 질문에서 작업 흐름으로 관점이 넓어지고 있습니다

처음에는 AI에게 좋은 답을 얻기 위해 질문을 잘 쓰는 것이 중요했습니다. 흔히 Prompt Engineering이라고 부르는 영역입니다. Claude Code처럼 파일을 읽고, 코드를 수정하고, 명령을 실행하는 도구를 쓰면 질문 하나만으로는 부족할 때가 있습니다.

이제는 “무슨 말을 할까”뿐 아니라 “어떤 맥락을 보여주고, 어떤 순서로 일하게 하고, 어떻게 검증할까”가 중요해집니다. 이 흐름을 넓게 보면 Agentic Engineering에 가깝습니다.

쉽게 말하면, 좋은 질문을 한 번 던지는 것보다 Claude Code가 어떤 순서로 코드를 읽고, 고치고, 검증하게 할지를 설계하는 쪽에 가깝습니다. 이번 뉴스레터는 이 변화를 거창한 개념보다, 팀에서 바로 써볼 수 있는 습관으로 풀어보려 합니다.

그래서 이번 8주 뉴스레터의 목표는 거창한 AI 이론을 설명하는 것이 아닙니다. 팀원이 쉬는 시간에 가볍게 읽고, 다음 작업에서 바로 써볼 수 있는 Claude Code 사용 습관을 하나씩 꺼내보는 것입니다. 매주 읽고 끝내는 글이 아니라, 써보고 괜찮았던 요청 문장이나 체크리스트를 작게 남겨보는 흐름으로 보겠습니다.

1. 흔한 시작: 바로 고쳐달라고 하기

가장 쉬운 요청은 바로 수정 요청입니다. 

상태 업데이트 버그 고쳐줘.
이 테스트 실패 고쳐줘.
이 서비스 코드 리팩토링해줘.

이 방식은 빠릅니다. 하지만 Claude Code가 원인을 충분히 보지 않고 그럴듯한 위치를 바로 고칠 수 있습니다. 그러면 처음에는 편해 보여도 리뷰할 때 변경 이유와 테스트 근거를 다시 확인하게 됩니다.

일반적인 요청

로그인 버그 고쳐줘.

조사부터 요청한 예시

로그인 실패 원인을 먼저 조사해줘.
관련 파일과 재현 흐름을 정리하고,
가장 작은 수정안을 제안한 뒤,
테스트 결과와 기존 기능에 영향을 줄 수 있는 부분까지 알려줘.

여기서 기존 기능에 영향을 줄 수 있는 부분이란, 이번 수정 때문에 같이 확인해야 하는 주변 동작을 뜻합니다. 예를 들어 로그인 로직을 고쳤다면, 정상 로그인뿐 아니라 비밀번호 오류, 계정 잠금, 세션 생성, 로그인 이력 저장 같은 흐름도 같이 볼 수 있습니다.

2. 좋은 요청은 길이가 아니라 순서가 다릅니다

Claude Code에게 일을 맡길 때는 긴 프롬프트보다 순서가 중요합니다. 바로 구현시키기 전에 먼저 봐야 할 것, 바꿔도 되는 범위, 확인하면 좋을 포인트를 알려주면 결과를 함께 검토하기 쉬워집니다. 

목표 정의
-> 원인 조사
-> 작은 수정안
-> 구현
-> 테스트
-> 수정 후 기존 기능 영향 확인

이 순서만 지켜도 Claude Code의 답변은 달라집니다. 코드만 나오는 것이 아니라, 변경 이유와 검증 근거가 같이 남기 때문입니다.

출력 형식까지 정해주면 더 좋습니다

작업 결과를 매번 비슷한 형식으로 받으면 리뷰하기 쉽습니다. 아래처럼 결과 형식까지 같이 요청할 수 있습니다. 

결과는 아래 형식으로 정리해줘.
1. 원인 후보
2. 관련 파일
3. 현재 동작 / 재현 흐름
4. 수정 방향
5. 테스트 방법
6. 기존 기능에 영향을 줄 수 있는 부분

3. 팀에 도움이 되는 사용법

팀에서 Claude Code를 쓰는 목적은 “AI가 코드를 많이 쓰게 하기”가 아닙니다. 우리에게 더 도움이 되려면 리뷰 부담을 줄이고, 변경 이유를 남기고, 다른 팀원도 이어서 이해할 수 있게 만드는 쪽이 중요합니다.

  • 관련 파일을 먼저 찾아보게 합니다.
  • 수정 전에 원인과 계획을 짧게 물어봅니다.
  • 가능하면 작은 변경부터 맡겨봅니다.
  • 테스트, 빌드, 재현 조건처럼 확인 가능한 근거를 같이 남겨봅니다.
  • 마지막에는 수정한 코드가 다른 기능에 영향을 주지 않는지 한 번 더 봅니다.

이 습관이 쌓이면 Claude Code는 개인 생산성 도구를 넘어 다음 사람이 이어보기 쉬운 작업 메모를 남기는 도구가 됩니다.

4. 참고해볼 요청 예시

작업 요청을 조금 더 정리해서 맡기고 싶을 때는 아래와 같은 문장을 참고해볼 수 있습니다.

참고해볼 요청 예시

아래는 상황에 맞게 줄이거나 바꿔서 사용할 수 있는 예시입니다.

이 작업은 바로 구현하지 말고 먼저 조사해줘.
관련 파일과 현재 동작을 정리하고,
수정 방향을 2가지 이하로 비교한 뒤,
테스트로 확인할 수 있는 방법까지 제안해줘.

핵심은 속도를 늦추자는 것이 아닙니다. 더 안전하게 작업하기 위해, 먼저 근거를 만들어두자는 것입니다.

이번 주 실천

버그 수정이나 리팩토링을 Claude Code에 맡길 때, 바로 “고쳐줘”라고 하기보다 먼저 관련 파일과 현재 동작을 정리해달라고 요청할 수도 있습니다. 원인이 명확하지 않은 작업에서는 리뷰할 때 필요한 정보를 먼저 얻는 데 도움이 될 수 있습니다. 괜찮았던 요청 문장이 있으면 팀에 짧게 공유해봅니다.

한 문장으로 정리

Claude Code를 사용할 때는 “빨리 만들어줘”뿐 아니라 “먼저 보고, 작게 고치고, 확인해서 알려줘”라는 방식도 함께 생각해볼 수 있습니다.
