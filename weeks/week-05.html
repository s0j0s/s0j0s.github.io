Week 5 · 반복 절차

Skill로 반복 설명
줄이기

매번 다시 쓰는 작업 순서와 답변 형식을
필요할 때 꺼내는 작은 레시피로 만듭니다.

키보드 방향키 또는 아래 버튼으로 넘겨보세요.

반복되는 요청

리뷰할 때마다 같은 기준을
다시 설명하고 있다면.

버그와 테스트 누락을 먼저 보고,
파일과 라인을 기준으로 짧게 써달라고 반복합니다.

Skill이 맡는 일

특정 작업의 절차를
필요할 때만 불러옵니다.

  1. 1언제 쓸까코드 리뷰, 테스트 계획, 이슈 조사
  2. 2어떤 순서일까확인할 항목과 제외할 작업
  3. 3무엇을 남길까답변 형식과 필요한 참고 자료

세션 시작에는 이름과 설명만 보이고, 본문은 Skill을 사용할 때 불러옵니다.

상시 기준과 작업 절차

항상 알아야 할 사실과
필요할 때 할 일을 나눕니다.

모든 작업에서 참고 CLAUDE.md

Java 8, Oracle 11g, 공통 명령과 팀 기준

특정 작업에서 실행 Skill

리뷰 순서, 테스트 체크리스트, 답변 형식

30줄짜리 절차가 CLAUDE.md에 쌓였다면 Skill 후보입니다.

가장 작은 구조

하나의 폴더와
SKILL.md로 시작합니다.

프로젝트 Skill .claude/skills/
backend-review/SKILL.md

팀이 함께 쓰고 저장소에서 관리

필요해지면 추가 references · scripts

긴 참고 자료, 반복 실행할 검사 도구

처음부터 큰 폴더를 만들 필요는 없습니다. 반복이 분명한 절차 하나면 충분합니다.

우리 팀의 첫 후보

익숙한 작업 하나부터
작게 시험합니다.

  1. 1backend-review트랜잭션, DB 조건, 테스트 누락
  2. 2junit-test-planner성공, 실패, 경계, 재시도 케이스
  3. 3issue-investigator호출 흐름, 원인 후보, 검증 방법

리뷰·테스트·조사를 하나의 거대한 Skill에 모두 넣지 않습니다.

실전 Skill 예시

설명보다
작업 순서가 보이게 씁니다.

---
description: Java 8 백엔드 변경을 리뷰할 때 사용
---

1. 트랜잭션 범위와 DB 업데이트 조건을 확인한다.
2. 예외 처리와 중복 처리 누락을 찾는다.
3. 빠진 JUnit 테스트를 제안한다.
4. 결과는 위험도 높은 순서로 짧게 쓴다.

두 가지 호출 방식

직접 실행하거나,
Claude가 골라 사용합니다.

사용자가 직접 /backend-review

실행 시점을 내가 정하고 싶을 때

요청에 맞춰 자동 선택 “현재 diff를 리뷰해줘”

description이 작업과 잘 맞을 때

배포나 커밋처럼 시점을 직접 정해야 한다면 disable-model-invocation: true로 자동 호출을 막습니다.

완성보다 반복 개선

작은 PR에서 써보고
쓸모없는 기준을 덜어냅니다.

  1. Step 1Skill 초안 하나 만들기
  2. Step 2작은 PR에서 직접 실행하기
  3. Step 3빠진 내용과 불필요한 의견 기록하기
  4. Step 42~3번 쓴 뒤 유용한 기준만 남기기

이번 주 실천

backend-review 하나를
작은 PR에 써봅니다.

한 번에 팀 표준으로 정하지 않습니다.
반복해서 도움이 된 기준만 함께 다듬습니다.

한 문장으로 정리

Skill은 반복 요청을
필요할 때 꺼내 쓰는
작은 작업 절차입니다.

다음 주에는 긴 작업을 별도 맥락의 SubAgent로 나눠봅니다.
