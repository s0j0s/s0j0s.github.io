Week 8

팀 플레이북: Claude Code를 팀의 개발 시스템으로 만들기

Claude Code 활용은 개인 생산성 팁에서 끝나지 않고, 팀의 작업 방식으로 정리될 때 가장 큰 효과가 납니다.

3줄 요약

  • Claude Code를 팀에 정착시키려면 개인 팁보다 공통 작업 방식이 필요합니다.
  • 플레이북은 요청 패턴, CLAUDE.md, Skills, MCP, SubAgent, Hooks의 기본값을 정리하는 문서입니다.
  • 처음에는 모든 것을 만들기보다 자주 반복되는 업무 하나를 골라 작게 표준화합니다.

이번 주 핵심

Claude Code를 잘 쓰는 팀은 AI 사용법을 개인 기량에 맡기지 않고, 팀의 개발 시스템으로 만듭니다.

지난 7주 동안 다룬 내용은 결국 하나로 이어집니다. Claude Code를 잘 쓰는 팀은 좋은 프롬프트 몇 개를 공유하는 데서 멈추지 않습니다. 일을 맡기는 루프를 정하고, 프로젝트 맥락을 정리하고, 반복 작업을 표준화하고, 외부 시스템과 안전장치를 연결합니다.

이것을 팀 플레이북으로 만들면 개인마다 다른 사용법이 줄어듭니다. 새로 합류한 사람도 “우리 팀은 Claude Code를 이렇게 쓴다”는 기준을 빠르게 이해할 수 있습니다.

1. 지난 7주 정리

  • 작업 루프: 바로 구현하지 않고 조사, 계획, 사람 승인, 구현, 테스트, 리뷰로 나눈다.
  • CLAUDE.md: 반복 설명하는 팀 규칙을 프로젝트 메모리로 관리한다.
  • Skills: 반복 작업을 재사용 가능한 작업 방식으로 고정한다.
  • MCP: GitHub, Jira, Confluence 같은 신뢰 가능한 외부 맥락을 연결한다.
  • SubAgent: 큰 작업을 조사, 구현, 리뷰 역할로 나눈다.
  • Hooks: 테스트, 정적 검사, 보안 검사 같은 기본 기준을 자동화한다.

작업 상태 업데이트 버그 사례로 보면 흐름은 하나입니다. Week 2의 기본 루프로 원인을 조사하고, Week 3의 CLAUDE.md로 트랜잭션 기준과 테스트 명령을 고정합니다. Week 4의 Skill은 리뷰와 테스트 보강 기준을 반복 적용하고, Week 5의 MCP는 이슈와 PR 논의, Confluence 문서를 읽게 합니다. Week 6의 SubAgent는 조사, 구현, 리뷰를 나누고, Week 7의 Hook은 관련 파일 변경 시 테스트나 경고를 붙입니다.

2. 팀 플레이북에 들어갈 내용

플레이북은 거창한 문서가 아니어도 됩니다. 중요한 것은 팀이 실제로 자주 하는 작업의 기본값을 정하는 것입니다.

  • 언제 Claude Code를 쓰는가
  • 어떤 요청 패턴을 기본으로 쓰는가
  • 어떤 작업은 사람이 반드시 리뷰하는가
  • 어떤 Skill을 공통으로 쓰는가
  • 어떤 MCP를 읽기 전용으로 허용하는가
  • 어떤 Hook을 경고, 자동 실행, 차단으로 둘 것인가
  • 쓰기 권한과 위험 명령은 어떻게 승인하는가

운영 기준은 구체적일수록 좋습니다. 예를 들어 GitHub PR, Jira 이슈, Confluence 문서 MCP는 읽기부터 허용하고, 댓글 작성, 이슈 수정, PR 업데이트는 승인 후에만 허용합니다. Hook은 관련 JUnit 테스트 경고, 정적 검사, secret 검사를 허용 목록으로 두고, 차단 Hook은 민감정보 유출이나 명확한 위험 명령처럼 합의가 쉬운 항목부터 시작합니다.

3. 팀 플레이북 템플릿

아래 템플릿을 그대로 복사해서 팀 문서의 초안으로 쓸 수 있습니다. 처음부터 완성하려고 하지 말고, 이번 달에 실제로 적용할 항목만 채워도 충분합니다. 

# Claude Code 팀 플레이북

## 1. 기본 작업 루프
- 리스크가 있는 작업은 조사 -> 계획 -> 사람 승인 -> 구현 -> 테스트 -> diff 리뷰 순서로 진행한다.
- 오타 수정, 단일 테스트 추가처럼 영향 범위가 작은 작업은 기본 루프를 줄일 수 있다.

## 2. 구현 전 승인 기준
- Claude가 수정할 파일 범위와 테스트 기준을 먼저 제안한다.
- 사람은 수정 범위가 과하지 않은지, 테스트 기준이 충분한지 확인한 뒤 구현을 승인한다.

## 3. CLAUDE.md 운영 기준
- 작업 시작 전 CLAUDE.md의 테스트 명령, 금지 범위, 리뷰 기준을 확인한다.
- 작업 후 문서와 실제 코드가 어긋나면 CLAUDE.md 개선 후보로 남긴다.

## 4. 공통 Skill
- 첫 Skill: backend-review
- 목적: diff에서 트랜잭션, DB 업데이트 조건, 예외 처리, 테스트 누락을 우선 확인한다.
- 운영: 실제 PR에 적용한 뒤 쓸모없는 의견과 빠진 기준을 반영한다.

## 5. MCP 권한 기준
- 읽기 허용: Jira 이슈, Confluence 문서, GitHub PR과 diff
- 승인 후 쓰기: Jira 댓글, PR 설명 수정, 리뷰 코멘트
- 별도 논의: 이슈 상태 변경, 운영 로그 조회, 배포 관련 작업

## 6. SubAgent 사용 기준
- 작은 수정은 기본 루프로 처리한다.
- 원인 분석이 길면 Investigator를 먼저 사용한다.
- 승인된 수정만 Builder에게 맡긴다.
- 구현 후에는 Reviewer가 실제 diff 기준으로 본다.

## 7. Hook 도입 순서
- 첫 Hook은 test-reminder로 시작한다.
- 처음에는 경고만 출력한다.
- 1~2주 사용 후 자동 실행이나 차단으로 올릴 항목만 고른다.

## 8. PR 전 확인 항목
- 변경 이유가 설명되어 있는가
- 실행한 테스트와 결과가 남아 있는가
- CLAUDE.md 기준을 어긴 부분은 없는가
- MCP나 외부 시스템에 쓰기 작업을 했다면 승인 기록이 있는가

4. 팀용 slash command 예시

팀에서 자주 쓰는 작업은 slash command로 묶어두면 사용성이 좋아집니다. 이름은 짧고, 동작은 팀 기준을 반영해야 합니다. 

/investigate
관련 파일, 호출 흐름, 원인 후보, 검증 방법을 조사한다. 구현은 하지 않는다.

/fix-with-tests
수정 계획을 먼저 제안하고, 승인 후 구현한다. 관련 테스트를 실행하고 결과를 요약한다.

/review
diff 기준으로 버그, 회귀 위험, 테스트 누락을 우선순위대로 리뷰한다.

/release-note
병합된 변경을 사용자 영향 중심으로 정리한다. 내부 구현 세부사항은 줄인다.

/issue-context
Jira, Confluence, GitHub 순서로 작업 맥락을 모은다. 쓰기 작업은 하지 않는다.

/simplify
동작을 바꾸지 않는 범위에서 복잡도를 낮출 후보를 찾고, 작은 단계로 제안한다.

5. 도입 전략

모든 것을 한 번에 만들 필요는 없습니다. 오히려 한 번에 도입하면 팀이 따라가기 어렵습니다. 반복되는 작업 하나를 고르고, 거기서부터 표준화하는 편이 좋습니다.

예를 들어 리뷰 품질이 문제라면 먼저 리뷰 Skill과 리뷰 요청 패턴을 만듭니다. 테스트 누락이 문제라면 Hooks와 /fix-with-tests부터 시작합니다. 이슈 맥락이 자주 빠진다면 GitHub, Jira, Confluence MCP부터 붙입니다.

6. 성공 기준

Claude Code 도입의 성공 기준을 코드 생성량으로 잡으면 방향이 흔들립니다. 더 많은 코드를 더 빨리 만드는 것보다, 더 적은 설명으로 더 안정적인 변경을 만들고, 리뷰와 테스트 루프를 단단하게 만드는 것이 중요합니다.

  • 리뷰에서 발견되는 기본 실수가 줄었는가
  • 테스트 실행과 결과 공유가 일관적인가
  • 반복 설명이 줄었는가
  • 새 팀원이 Claude Code 사용 방식을 빠르게 따라오는가
  • 위험한 작업에서 승인과 검증 절차가 지켜지는가

7. 첫 달 적용 체크리스트

처음 한 달은 도구를 많이 붙이는 기간이 아니라, 팀원이 같은 방식으로 쓰기 시작하는 기간입니다. 아래 항목만 채워도 충분히 출발할 수 있습니다. 

1주차: 기본 요청 문장 하나를 팀 채널에 공유한다.
2주차: 조사 -> 계획 -> 사람 승인 -> 구현 -> 테스트 -> diff 리뷰 루프를 실제 이슈 하나에 적용한다.
3주차: CLAUDE.md 개선 후보를 하나 남긴다.
4주차: backend-review Skill 초안을 만들고 작은 PR 하나에 써본다.
5주차: MCP 읽기/쓰기 권한 기준을 세 칸으로 정리한다.
6주차: SubAgent를 쓸 작업과 쓰지 않을 작업 기준을 정한다.
7주차: test-reminder Hook 후보 문구를 정한다.
8주차: 위 내용을 팀 플레이북 초안으로 묶는다.

팀에서 합의하면 좋은 기준

Claude Code 플레이북은 완성된 문서가 아니라 운영 중 계속 다듬는 시스템입니다. 반복되는 불편이 보이면 메모리, Skill, MCP, Hook 중 어디에 반영할지 정합니다.

이번 주 실천

플레이북을 완성하려고 하지 말고, 이번 달에 가장 자주 반복된 작업 하나를 골라 요청 패턴과 검증 기준부터 적어봅니다.

마무리

Claude Code는 코드를 대신 써주는 도구로도 쓸 수 있습니다. 하지만 팀 차원의 효과는 작업 루프, 맥락, 반복 작업, 외부 연결, 역할 분리, 안전장치가 함께 맞물릴 때 나옵니다. 결국 중요한 질문은 “Claude에게 무엇을 시킬까”가 아니라 “우리 팀의 개발 시스템 안에서 Claude가 어떤 역할을 맡을까”입니다.
