Week 7 · 자동 확인

Hooks로 잊기 쉬운 확인
알려주기

알림부터 시작해 자동 검사와
정말 위험한 작업의 차단으로 확장합니다.

키보드 방향키 또는 아래 버튼으로 넘겨보세요.

대화로만 반복하면

“테스트 확인해줘”를
매번 기억해야 합니다.

사람도 Claude도 반복 확인을 놓칠 수 있습니다.
변경 속도가 빨라질수록 검증 루프가 뒤처집니다.

Hooks가 맡는 일

정해둔 시점에
빠짐없이 실행됩니다.

  1. 1알려주기관련 테스트와 호환성 확인 추가
  2. 2실행하기테스트, 정적 검사, secret 검사
  3. 3차단하기위험 명령과 명확한 정책 위반

Claude가 실행할지 판단하는 지침과 달리 Hook의 이벤트는 정해진 조건에서 동작합니다.

파일과 등록 위치

스크립트만 두면
아직 Hook이 아닙니다.

언제 실행할지 등록 .claude/settings.json

이벤트, matcher, 실행할 handler

무엇을 할지 구현 .claude/hooks/
test-reminder.sh

알림, 테스트, 검사, 차단 판단

.claude/hooks/는 팀이 정한 스크립트 위치일 뿐, 파일만 만들어서는 실행되지 않습니다.

실행 전과 실행 후

막아야 한다면
도구가 실행되기 전이어야 합니다.

실행 전 PreToolUse

도구 입력을 검사하고 위험한 호출을 차단

실행 후 PostToolUse

결과에 피드백과 추가 컨텍스트 전달

PostToolUse 시점에는 이미 도구가 실행되어 이전 행동을 되돌릴 수 없습니다.

첫 Hook은 가볍게

파일 수정 뒤
확인할 내용을 더합니다.

{
  "hooks": {"PostToolUse": [{
    "matcher": "Write|Edit",
    "hooks": [{
      "type": "command",
      "command": "${CLAUDE_PROJECT_DIR}/.claude/hooks/test-reminder.sh"
    }]
  }]}
}

리마인더 출력

일반 문자열 대신
구조화된 결과를 돌려줍니다.

#!/usr/bin/env bash

printf '%s\n' \
'{"hookSpecificOutput":{"hookEventName":"PostToolUse",
"additionalContext":"관련 JUnit 테스트와 Java 8, Oracle 11g 호환성을 확인하세요."}}'

exit 0

차단은 제한적으로

명확한 위험만
exit 2로 멈춥니다.

  • PreToolUse에서 실행할 명령과 인자를 검사
  • 차단 사유는 stderr로 Claude에게 전달
  • secret 노출, 강제 push, 운영 설정 삭제처럼 범위를 좁힘
  • 조직 전체 정책은 회사의 managed settings를 우선

일반적인 설계 판단이나 배포 리스크 수용은 Hook이 아니라 사람이 결정합니다.

우리 팀의 첫 후보

자주 빠뜨리지만
판단이 적은 확인부터 시작합니다.

  1. 1test-reminder관련 JUnit 테스트와 결과 확인
  2. 2secret-guard토큰, 비밀번호, 인증키 패턴
  3. 3oracle11g-sql-guard지원하지 않는 SQL 문법 확인

오탐이 많은 검사는 알림으로 낮추고, 명확한 패턴만 차단 후보로 둡니다.

팀에 적용하는 순서

알림으로 관찰한 뒤
필요한 자동화만 올립니다.

  1. Step 1놓치는 확인 하나 고르기
  2. Step 2알림으로 1~2주 사용하기
  3. Step 3오탐과 실행 시간을 확인하기
  4. Step 4자동 실행과 차단 후보만 남기기

이번 주 실천

test-reminder 하나를
알림 단계로 써봅니다.

팀이 불편하지 않은지 확인한 뒤
제한된 테스트 자동 실행을 검토합니다.

한 문장으로 정리

Hooks는 기억을 대신하되,
사람의 판단까지
대신하지 않습니다.
