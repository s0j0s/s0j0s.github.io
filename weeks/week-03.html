Week 3 · 팀 설정

우리 팀의 Claude Code 설정,
무엇부터 쓸까

매번 다시 설명하지 않도록
팀의 매뉴얼, 권한, 도구를 나눠봅니다.

키보드 방향키 또는 아래 버튼으로 넘겨보세요.

Week 1·2에서 이어서

좋은 요청과 계획을
매번 다시 설명하고 있다면.

1주차의 작업 순서와 2주차의 완료 기준 중 반복되는 내용을 남겨두면,
Claude Code는 다음 작업에서도 같은 팀 기준을 참고합니다.

하나의 비유로 이해하기

Claude Code 설정은
새 팀원을 위한 업무 안내서입니다.

반복해서 알려줄 팀 기준과 실행 범위,
필요할 때 꺼내 쓸 도구를 따로 관리합니다.

먼저 큰 그림

복잡해 보여도 이번 주에는
세 묶음만 살펴봅니다.

  1. 01팀 안내프로젝트와 일하는 방식
  2. 02실행 설정허용·확인·차단할 행동
  3. 03확장 도구특정한 불편을 해결하는 기능

파일 이름을 외우기 전에 각 항목이 무슨 일을 하는지부터 봅니다.

1 · 프로젝트 팀 안내

CLAUDE.md에는
매번 필요한 설명을 담습니다.

  1. 1프로젝트 설명서Claude Code가 작업할 때 참고하는 팀의 업무 안내
  2. 2반복 설명 줄이기환경과 작업 방식을 매번 처음부터 설명하지 않도록
  3. 3작업 기준 전달빌드·테스트·구조·리뷰 기준을 작업 전에 공유

Java 8, Oracle 11g, 테스트 명령처럼 모든 작업에서 자주 필요한 기준부터 남깁니다.

우리 팀처럼 두 파일을 함께 쓴다면

공식적으로 역할이 다른
두 파일은 아닙니다.

팀에서 정한 역할 · 프로젝트 입구 ./CLAUDE.md

핵심 환경, 공통 명령, 전체 구조처럼 먼저 볼 기준

팀에서 정한 역할 · Claude Code 세부 안내 ./.claude/CLAUDE.md

Claude Code 작업 흐름, 검토 방법, 세부 운영 기준

두 파일 모두 프로젝트 업무 매뉴얼입니다. 입구와 세부 안내로 나눈 것은 우리 팀의 약속입니다. /context에서 두 파일이 실제 컨텍스트에 포함됐는지 확인합니다.

공용 매뉴얼과 개인 메모

함께 볼 내용과
나만 볼 내용을 나눕니다.

팀과 공유 CLAUDE.md

프로젝트 환경, 공통 명령, 설계와 리뷰 기준

개인 환경에만 적용 CLAUDE.local.md

개인 작업 습관, 로컬 메모, 공유하기 애매한 선호

팀 기준에 개인 취향이 섞이지 않도록 나눕니다. 개인 파일은 저장소에 커밋하지 않습니다.

상황별 세부 지침

.claude/rules/는 경로를 지정하면
필요할 때만 불러옵니다.

한 파일에 모두 작성 CLAUDE.md
Java·SQL·테스트·문서 규칙

관련 없는 작업에서도 모든 설명을 함께 참고

상황별 규칙으로 분리 .claude/rules/java.md
.claude/rules/database.md
.claude/rules/testing.md

관련 파일을 다룰 때 필요한 규칙을 중심으로 참고

paths를 지정하지 않은 규칙은 항상 참고합니다. 특정 파일에만 필요한 규칙에는 적용 경로도 적습니다.

2 · 실제 권한과 동작

settings.json은
부탁이 아니라 실행 범위를 정합니다.

팀과 공유 .claude/settings.json

팀 공용 권한, 도구 설정, 읽지 못하게 할 파일

개인 환경에만 적용 .claude/settings.local.json

개인 실험, 로컬 환경 차이, 저장소에 커밋하지 않을 설정

CLAUDE.md가 “이렇게 일해 주세요”라는 안내라면, settings는 어떤 행동을 허용·확인·차단할지 정합니다.

우리 팀의 AWS Bedrock 환경

설정은 공유해도,
비밀 값은 공유하지 않습니다.

  • 회사에서 관리하는 인증과 보안 정책은 그대로 따르기
  • 인증 정보와 토큰을 저장소 파일에 넣지 않기
  • 권한 설정에서 비밀 정보 경로를 읽지 못하게 했는지 확인하기
  • 프로젝트 설정으로 회사 정책을 우회하지 않기

공유 설정에는 연결 방법만 남깁니다. 실제 비밀 값은 회사가 정한 방식으로 관리합니다.

3 · 필요할 때 꺼내는 도구 상자

불편의 종류에 맞춰
하나씩 선택합니다.

  1. 1외부 정보 · MCPGitHub·Jira·Confluence의 정보가 필요할 때
  2. 2반복 절차 · Skill같은 절차를 현재 작업에서 반복할 때
  3. 3독립 작업 · SubAgent긴 조사나 분석을 별도 컨텍스트에 맡길 때
  4. 4자동 실행 · Hooks특정 시점에 반드시 실행하거나 차단할 때

전부 갖출 필요는 없습니다. 다음 주부터 각 도구가 필요한 순간을 하나씩 살펴봅니다.

Week 1·2의 기준을 팀 설정으로

반복해서 말하는 내용만
알맞은 자리에 남깁니다.

  1. 1“수정 전 관련 파일부터 확인”항상 적용할 팀 기준이라면 CLAUDE.md
  2. 2“API 수정에는 이 테스트까지”특정 경로에만 필요하다면 .claude/rules/
  3. 3“이 명령은 실행 전 확인”실제 허용 범위라면 settings.json

조사→계획→검토처럼 반복되는 여러 단계는 길게 쌓지 말고 필요할 때 불러오는 Skill로 나눕니다.

처음에는 두 가지면 충분합니다

팀 안내와 권한부터,
도구는 필요할 때 더합니다.

  1. Step 1/init으로 CLAUDE.md 초안 만들고 팀 기준에 맞게 다듬기
  2. Step 2settings.json에서 권한과 실행 범위 확인하기
  3. Step 3길어진 기준은 .claude/rules/로 나누기
  4. Step 4반복되는 불편이 생기면 맞는 도구 더하기

CLAUDE.md는 짧게 유지하고 담당자를 정해 코드처럼 검토합니다. 도구는 반복되는 불편이 생길 때 하나씩 더합니다.

한 문장으로 정리

반복할 팀 기준은 CLAUDE.md에,
실행 범위는
settings.json에 둡니다.
