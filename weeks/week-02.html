Week 2

조사, 계획, 테스트까지 이어지는 작업 루프

약 4분 · 실무 활용 팁

Week 1에서 살펴본 “먼저 조사해줘”를 바탕으로, 조사 결과를 계획, 구현, 테스트, 변경 영향 검토까지 연결하는 흐름을 정리합니다.

3줄 요약

  • “바로 고쳐줘”는 작은 작업에는 편하지만, 복잡한 작업에서는 원인과 수정 범위를 다시 확인해야 할 수 있습니다.
  • 조사 결과에서 수정 범위와 테스트 방법을 먼저 정하면, 구현 후 계획대로 바뀌었는지 확인하기 쉽습니다.
  • 이번 글에서는 버그 수정을 맡길 때 이 흐름을 어떻게 활용할 수 있는지 살펴봅니다.

1주차에서는 조사부터 요청하는 방법을 살펴봤습니다. 이번에는 조사 결과에서 수정 범위와 테스트 방법을 정하고, 구현 결과가 계획과 맞는지 확인하는 흐름을 살펴봅니다.

이번 글의 예시 버그

작업 상태 업데이트 API가 성공 응답을 반환했지만, DB의 상태는 바뀌지 않은 상황을 기준으로 보겠습니다. 상태 전환 조건, DB 업데이트 결과, 예외 처리를 함께 확인해야 하는 가상의 Java 백엔드 버그입니다.

1. 조사 결과를 보고, 어디까지 고칠지 정하기

Claude Code가 조사 결과를 보여주면, 바로 구현을 맡기기 전에 수정 범위와 테스트 항목을 확인해볼 수 있습니다. 아래 예시에서는 세 단계로 나누어 살펴보겠습니다.

1 Claude Code가 정리

조사 결과 확인

  • 현재 상태가 UPDATE 조건과 맞지 않아 변경된 행이 0건입니다.
  • 변경된 행이 0건이어도 API는 성공 응답을 반환합니다.
  • 이 상황을 확인하는 테스트가 없습니다.
2 사람이 범위를 확인

수정 범위 확인

  • 변경된 행이 0건이면 성공이 아니라 실패로 처리합니다.
  • 다른 기능은 건드리지 않고 상태 업데이트 로직만 수정합니다.
3 구현 후 함께 확인

테스트 항목 확인

  • 허용된 상태 전환이 정상적으로 처리되는지 확인합니다.
  • 허용되지 않은 상태 전환이 실패로 처리되는지 확인합니다.

계획을 확인한 뒤 이어서 요청하기

세 단계의 내용이 괜찮다면 아래처럼 구현을 이어서 요청할 수 있습니다.

위에서 정한 수정 범위대로 구현하고,
테스트 결과와 추가로 확인할 부분을 정리해줘.

이렇게 나누어두면 무엇을 고칠지와 어떻게 확인할지가 분명해져, 구현 후에도 계획한 범위대로 수정됐는지 검토하기 쉽습니다.

2. 계획을 확인하면 달라지는 점

조사 결과를 바로 구현하는 경우와 변경 계획을 먼저 확인하는 경우를 비교해보겠습니다.

바로 구현을 맡긴 경우

  • 수정 속도는 빠를 수 있습니다.
  • 필요한 범위보다 수정 범위가 넓어질 수 있습니다.
  • 테스트 근거가 부족하면 리뷰 부담이 커집니다.

계획을 먼저 확인한 경우

  • 확인 단계가 추가되므로 작은 작업에서는 오히려 번거로울 수 있습니다.
  • 수정할 파일과 제외할 범위가 먼저 보입니다.
  • 구현 전에 테스트 방법을 확인할 수 있습니다.
  • 수정한 코드가 기존 기능에 어떤 영향을 줄지 확인하기 쉽습니다.

구현 후 리뷰에서 다시 확인하게 되는 질문

리뷰에서는 왜 이 파일을 바꿨는지, 다른 상태값에도 영향이 있는지, 어떤 테스트로 확인했는지를 살펴보게 됩니다. 구현 전 계획에 이 질문을 포함하고 작업 후 실제 결과를 덧붙이면, 리뷰에서 같은 내용을 다시 확인하는 부담을 줄일 수 있습니다.

3. 기본 작업 루프

목표 정의
-> 코드 조사
-> 변경 계획
-> 계획 확인
-> 구현
-> 테스트
-> 변경 영향 검토

목표 정의에서는 원하는 결과와 제약을 말합니다. 코드 조사에서는 Claude Code가 관련 파일, 호출 흐름, 기존 테스트를 찾습니다. 변경 계획에서는 수정 범위와 검증 방법을 정리합니다. 계획 확인에서는 수정할 파일과 테스트 방법이 적절한지 사람이 짧게 봅니다. 구현과 테스트는 확인한 범위 안에서 진행하도록 요청할 수 있습니다.

여기서 계획 확인은 거창한 승인 절차를 말하는 것이 아닙니다. “이 파일들만 바꾸고, 이 테스트로 확인하자”는 정도를 사람이 한 번 보는 단계입니다. 이 단계를 거치면 Claude Code의 수정 범위가 불필요하게 넓어지거나, 테스트 방법 없이 구현부터 진행하는 일을 줄이는 데 도움이 됩니다.

테스트가 끝난 뒤에는 Claude Code에게 수정한 코드를 기준으로 영향을 받을 수 있는 주변 동작도 확인해달라고 요청할 수 있습니다. “무엇을 바꿨고, 어떤 동작이 영향을 받을 수 있으며, 어떤 테스트로 확인했는가”를 정리해달라고 하면 리뷰할 때 확인할 기준이 생깁니다.

변경 후에 한 번 더 확인할 것

코드를 수정한 뒤에는 영향을 받을 수 있는 주변 기능이 있는지 확인해볼 수 있습니다. 테스트가 빠진 부분은 없는지, 계획보다 수정 범위가 넓어지지 않았는지도 같이 보면 좋습니다.

4. 언제는 바로 맡겨도 될까

모든 작업에 긴 루프가 필요한 것은 아닙니다. 오타 수정, 주석 정리, 로그 문구 변경처럼 영향 범위가 작은 작업은 바로 맡겨도 괜찮습니다. 반대로 다음과 같은 작업에서는 조사와 계획을 먼저 받아보는 방식도 도움이 됩니다.

  • 수정 파일이 여러 영역에 걸치거나 기존 설계 의도를 알아야 한다.
  • 트랜잭션 범위, DB 업데이트 조건, 상태 전환처럼 영향 범위가 크다.
  • 배치 처리, 외부 시스템 연동, 예외 처리, 재시도 로직에 영향이 있다.
  • 테스트 실패의 원인이 명확하지 않거나 판단이 필요한 리팩토링이다.

참고해볼 요청 예시

버그 수정이나 테스트 실패를 맡길 때 상황에 맞게 줄이거나 바꿔서 사용할 수 있습니다.

지금은 코드를 수정하지 말고 관련 파일과 현재 동작을 조사해줘.
원인 후보, 수정 범위, 테스트 방법을 정리한 뒤 멈춰줘.
내가 계획을 확인하면 그 범위에서 구현하고 테스트해줘.
실행하지 못한 테스트나 추가로 확인할 영향 범위가 있으면 함께 알려줘.

한 문장으로 정리

조사한 내용에서 수정 범위와 테스트 방법을 정한 뒤 구현하면, 변경 이유와 확인 결과를 함께 검토하기 쉬워집니다.
