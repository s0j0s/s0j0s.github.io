단순한 화면 뒤의 질문

버튼은 동작하지만,
정해야 할 규칙이 남았습니다.

  1. 1변경 순서배송 완료 주문도 취소할 수 있나요?
  2. 2변경 담당자상담원과 운영자 모두 변경할 수 있나요?
  3. 3동시 저장두 사람이 동시에 저장하면 어떻게 되나요?
  4. 4기록 남기기누가 언제 바꿨는지 기록해야 하나요?

에이전트가 구현을 돕고 사람이 범위와 품질을 확인하는 방식을 Agentic Engineering이라고 합니다.

Step 1 · Claude Code가 초안 작성

Claude Code가
PLAN.md 초안을 작성합니다.

  1. 1주문 조회·상태 변경 화면
  2. 2바꿀 수 있는 순서·변경 담당자
  3. 3저장 기록·문제 상황 테스트 기준

Claude Code가 만든 계획은 정답이 아니라, 구현 전에 사람이 함께 고치는 초안입니다.

Step 2 · 사람이 계획을 확인

필요한 것은 더하고,
이번 범위는 선명하게.

이번에 포함 변경 순서·담당자
기록 남기기·두 번 저장 막기
이번에는 제외 고객 알림·여러 주문 변경
자동 환불

원하는 기능뿐 아니라, 이번에 만들지 않을 기능도 함께 확인합니다.

Step 3 · 구현 순서를 작은 단계(Phase)로 나누기

한 번에 만들지 않고,
확인 가능한 단위로 나눕니다.

  1. Phase 01조회와 변경 화면
  2. Phase 02변경 순서와 담당자
  3. Phase 03저장과 기록
  4. 마지막 확인동시 저장·오류 확인

phase-01-order-ui.md처럼 단계별 문서로 나누면 각 범위와 완료 조건을 따로 확인합니다.

Step 4 · Phase마다 사람이 확인

하나씩 만들고,
확인 후 다음으로.

  1. 01Phase 구현정한 범위만 작업
  2. 02결과 정리구현 내용과 테스트
  3. 03사람이 확인고치거나 계속할지 결정
  4. 04다음 Phase확인한 뒤에 시작

↩ 수정이 필요하면 같은 Phase로 돌아갑니다.

한 Phase를 구현한 뒤 결과를 정리하고 멈춰줘.
내가 확인하면 다음 Phase를 진행해줘.

확인한 계획과 구현·테스트 결과를 비교

계획이 필요한 작업에 꺼내 쓰기

이번 작업에 바로 써볼 요청

지금은 구현하지 말고, 요청한 기능의 화면·동작·완료 기준을
PLAN.md에 정리해줘. 불분명한 선택지는 따로 표시하고 멈춰줘.
내가 계획을 확인하면 작업을 확인 가능한 Phase로 나눠줘.
Phase별 범위와 완료 기준을 다시 확인한 뒤 순서대로 구현해줘.
각 Phase가 끝나면 구현 내용과 테스트 결과를 정리하고 멈춰줘.
내가 확인한 뒤 다음 Phase를 진행해줘.

오타 수정처럼 범위와 완료 기준이 분명한 작업은 바로 맡겨도 괜찮습니다.

한 문장으로 정리

계획을 함께 다듬고 나누어 구현하면,
원하는 결과와 실제 결과를
비교하기 쉬워집니다.
