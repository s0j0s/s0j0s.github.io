Week 8

팀에서 편하게 꺼내 쓰는 Claude Code 메모

약 6분 · 실무 활용 팁

각자 써본 Claude Code 활용법을 작은 메모로 남겨두면, 다음에 비슷한 작업을 할 때 더 편하게 꺼내 쓸 수 있습니다.

3줄 요약

  • Claude Code 활용법은 개인 팁으로만 두기보다 쓸 만한 것만 가볍게 공유해도 도움이 됩니다.
  • 팀 메모에는 요청 패턴, CLAUDE.md, Skills, MCP, SubAgent, Hooks에서 써본 내용을 짧게 남깁니다.
  • 처음에는 모든 것을 만들기보다 자주 반복되는 업무 하나만 골라 작게 시도합니다.

이번 주 포인트

Claude Code를 잘 쓰는 팀은 거창한 규칙보다, 써보고 괜찮았던 요청과 흐름을 다음 사람이 참고할 수 있게 남깁니다.

지난 7주 동안 다룬 내용은 결국 하나로 이어집니다. Claude Code를 편하게 쓰려면 좋은 프롬프트 몇 개만 공유해도 도움이 됩니다. 여기에 일을 맡기는 흐름, 프로젝트 메모, 반복 작업, 외부 시스템, 리마인더를 조금씩 붙여볼 수 있습니다.

이것을 작은 팀 메모로 남겨두면 각자 써본 방법을 다시 찾기 쉽습니다. 새로 합류한 사람도 “이 팀에서는 이런 식으로 써봤구나” 정도를 빠르게 참고할 수 있습니다.

1. 지난 7주 정리

  • 작업 루프: 바로 구현하지 않고 조사, 계획, 방향 확인, 구현, 테스트, 리뷰로 나눠본다.
  • CLAUDE.md: 반복 설명하는 내용을 프로젝트 메모로 남긴다.
  • Skills: 반복 작업을 다시 꺼내 쓸 수 있게 남긴다.
  • MCP: GitHub, Jira, Confluence 같은 외부 맥락을 필요할 때 연결한다.
  • SubAgent: 큰 작업을 조사, 구현, 리뷰 역할로 나눠본다.
  • Hooks: 테스트, 정적 검사, 보안 검사처럼 잊기 쉬운 확인을 알려준다.

작업 상태 업데이트 버그 사례로 보면 흐름은 하나입니다. Week 2의 기본 루프로 원인을 조사하고, Week 3의 CLAUDE.md로 테스트 명령과 주의할 점을 확인합니다. Week 4의 Skill은 리뷰와 테스트 보강 요청을 반복해서 쓰기 쉽게 만들고, Week 5의 MCP는 이슈와 PR 논의, Confluence 문서를 읽게 합니다. Week 6의 SubAgent는 조사, 구현, 리뷰를 나눠보고, Week 7의 Hook은 관련 파일 변경 시 테스트 알림을 붙입니다.

2. 팀 메모에 들어갈 내용

팀 메모는 거창한 문서가 아니어도 됩니다. 중요한 것은 실제로 자주 하는 작업에서 다시 쓰고 싶은 요청과 확인 포인트를 남기는 것입니다.

  • 언제 Claude Code를 쓰는가
  • 어떤 요청 패턴이 편했는가
  • 어떤 작업은 사람이 한 번 더 보는가
  • 어떤 Skill을 같이 써봤는가
  • 어떤 MCP를 읽기 용도로 써봤는가
  • 어떤 Hook은 알림으로 두고, 어떤 것은 자동 실행해볼 수 있는가
  • 쓰기 권한과 위험 명령은 언제 한 번 더 확인하는가

처음 메모는 구체적이면 충분합니다. 예를 들어 GitHub PR, Jira 이슈, Confluence 문서 MCP는 읽기부터 써보고, 댓글 작성, 이슈 수정, PR 업데이트는 사람이 한 번 확인한 뒤 맡깁니다. Hook은 관련 JUnit 테스트 알림, 정적 검사, secret 검사처럼 부담이 적은 항목부터 시작합니다.

3. 팀 메모 템플릿

아래 템플릿을 그대로 복사해서 팀 메모의 초안으로 쓸 수 있습니다. 처음부터 완성하려고 하지 말고, 이번 달에 실제로 적용할 항목만 채워도 충분합니다. 

# Claude Code 팀 메모

## 1. 써볼 만한 작업 루프
- 리스크가 있는 작업은 조사 -> 계획 -> 방향 확인 -> 구현 -> 테스트 -> diff 리뷰 순서로 맡겨본다.
- 오타 수정, 단일 테스트 추가처럼 영향 범위가 작은 작업은 바로 맡겨도 괜찮다.

## 2. 구현 전 방향 확인
- Claude가 수정할 파일 범위와 테스트 방법을 먼저 제안하게 해본다.
- 사람은 수정 범위가 과하지 않은지, 테스트 방법이 충분한지 한 번 본다.

## 3. CLAUDE.md 참고 메모
- 작업 시작 전 CLAUDE.md의 테스트 명령, 조심할 범위, 리뷰 포인트를 확인한다.
- 작업 후 문서와 실제 코드가 어긋나면 CLAUDE.md 개선 후보로 남긴다.

## 4. 같이 써볼 Skill
- 첫 Skill: backend-review
- 목적: diff에서 트랜잭션, DB 업데이트 조건, 예외 처리, 테스트 누락을 우선 확인한다.
- 운영: 실제 PR에 적용한 뒤 쓸모없는 의견과 빠진 내용을 반영한다.

## 5. MCP 권한 메모
- 읽기부터 써보기: Jira 이슈, Confluence 문서, GitHub PR과 diff
- 확인 후 쓰기: Jira 댓글, PR 설명 수정, 리뷰 코멘트
- 별도 논의: 이슈 상태 변경, 운영 로그 조회, 배포 관련 작업

## 6. SubAgent를 써볼 만한 경우
- 작은 수정은 기본 루프로 처리한다.
- 원인 분석이 길면 Investigator를 먼저 써본다.
- 사람이 확인한 수정만 Builder에게 맡긴다.
- 구현 후에는 Reviewer가 실제 diff 기준으로 본다.

## 7. Hook 도입 순서
- 첫 Hook은 test-reminder로 시작한다.
- 처음에는 알림만 출력한다.
- 1~2주 사용 후 자동 실행이나 실행 중단으로 올릴 항목만 고른다.

## 8. PR 전 확인 항목
- 변경 이유가 설명되어 있는가
- 실행한 테스트와 결과가 남아 있는가
- CLAUDE.md와 어긋나는 부분은 없는가
- MCP나 외부 시스템에 쓰기 작업을 했다면 사람이 확인했는가

4. 팀용 slash command 예시

팀에서 자주 쓰는 작업은 slash command로 묶어두면 사용성이 좋아집니다. 이름은 짧고, 동작은 실제로 자주 쓰는 흐름을 반영하면 충분합니다. 

/investigate
관련 파일, 호출 흐름, 원인 후보, 검증 방법을 조사한다. 구현은 하지 않는다.

/fix-with-tests
수정 계획을 먼저 제안하고, 방향 확인 후 구현한다. 관련 테스트를 실행하고 결과를 요약한다.

/review
diff 기준으로 버그, 회귀 위험, 테스트 누락을 우선순위대로 리뷰한다.

/release-note
병합된 변경을 사용자 영향 중심으로 정리한다. 내부 구현 세부사항은 줄인다.

/issue-context
Jira, Confluence, GitHub 순서로 작업 맥락을 모은다. 쓰기 작업은 하지 않는다.

/simplify
동작을 바꾸지 않는 범위에서 복잡도를 낮출 후보를 찾고, 작은 단계로 제안한다.

5. 도입 전략

모든 것을 한 번에 만들 필요는 없습니다. 오히려 한 번에 도입하면 팀이 따라가기 어렵습니다. 반복되는 작업 하나를 고르고, 거기서부터 작게 써보는 편이 좋습니다.

예를 들어 리뷰 품질이 문제라면 먼저 리뷰 Skill과 리뷰 요청 패턴을 만듭니다. 테스트 누락이 문제라면 Hooks와 /fix-with-tests부터 시작합니다. 이슈 맥락이 자주 빠진다면 GitHub, Jira, Confluence MCP부터 붙입니다.

그대로 써봐도 되는 요청

팀 메모 초안을 만들 때 사용할 수 있습니다.

우리 팀의 Claude Code 메모 초안을 만들어줘.
대상은 Java 8 백엔드 개발자이고,
자주 하는 작업은 버그 조사, 작은 수정, 리뷰, 테스트 보강이야.
요청 패턴, CLAUDE.md에 넣을 내용, 공통 Skill 후보,
GitHub/Jira/Confluence MCP 사용 메모,
Hooks로 알려줄 항목을 짧게 정리해줘.

6. 잘 쓰고 있는지 보는 힌트

Claude Code를 잘 쓰고 있는지 보는 척도를 코드 생성량으로 잡으면 방향이 흔들립니다. 더 많은 코드를 더 빨리 만드는 것보다, 더 적은 설명으로 더 안정적인 변경을 만들고, 리뷰와 테스트 루프를 편하게 남기는 것이 중요합니다.

  • 리뷰에서 발견되는 기본 실수가 줄었는가
  • 테스트 실행과 결과 공유가 조금 더 쉬워졌는가
  • 반복 설명이 줄었는가
  • 새 팀원이 Claude Code 사용 방식을 빠르게 따라오는가
  • 위험한 작업에서 사람이 한 번 더 확인하고 넘어가는가

7. 첫 달에 가볍게 해볼 것

처음 한 달은 도구를 많이 붙이는 기간이 아니라, 부담 없이 몇 가지를 써보는 기간입니다. 아래 항목 중 한두 개만 해봐도 충분히 출발할 수 있습니다. 

1주차: 써볼 만한 요청 문장 하나를 팀 채널에 공유한다.
2주차: 조사 -> 계획 -> 방향 확인 -> 구현 -> 테스트 -> diff 리뷰 흐름을 실제 이슈 하나에 써본다.
3주차: CLAUDE.md 개선 후보를 하나 남긴다.
4주차: backend-review Skill 초안을 만들고 작은 PR 하나에 써본다.
5주차: MCP 읽기/쓰기 권한 범위를 세 칸으로 메모한다.
6주차: SubAgent를 써볼 만한 작업과 쓰지 않아도 되는 작업을 적어본다.
7주차: test-reminder Hook 후보 문구를 하나 써본다.
8주차: 위 내용을 작은 팀 메모로 묶는다.

팀에서 가볍게 맞춰보면 좋은 것

Claude Code 팀 메모는 완성된 문서가 아니라 써보면서 조금씩 다듬는 기록입니다. 반복되는 불편이 보이면 메모리, Skill, MCP, Hook 중 어디에 남기면 좋을지 이야기해봅니다.

이번 주 실천

팀 메모를 완성하려고 하지 말고, 이번 달에 가장 자주 반복된 작업 하나를 골라 요청 패턴과 확인 포인트부터 적어봅니다.

마무리

Claude Code는 코드를 대신 써주는 도구로도 쓸 수 있습니다. 하지만 팀 차원의 효과는 작업 루프, 맥락, 반복 작업, 외부 연결, 역할 분리, 리마인더가 함께 맞물릴 때 나옵니다. 결국 중요한 질문은 “Claude에게 무엇을 시킬까”가 아니라 “우리 팀에서는 Claude를 어떤 순간에 쓰면 편할까”입니다.
