Week 2 of 8 작업 루프

Week 2

조사, 계획, 테스트까지 이어지는 작업 루프

Week 1에서 살펴본 “먼저 조사해줘”를 바탕으로, 조사 결과를 계획, 구현, 테스트, 수정 후 확인까지 연결하는 흐름을 정리해봅니다.

3줄 요약

  • “바로 고쳐줘”는 빠르지만 조사 근거와 수정 범위가 흐려지기 쉽습니다.
  • 조사, 계획, 방향 확인, 구현, 테스트, 수정 후 확인 순서로 맡기면 리뷰할 근거가 남습니다.
  • 이번 글에서는 버그 수정을 맡길 때 이 흐름을 어떻게 활용할 수 있는지 살펴봅니다.

이번 주 포인트

Week 1에서는 바로 구현을 맡기기보다 먼저 조사하게 하는 방식을 살펴봤습니다. Week 2에서는 그 흐름을 조금 확장해서, 조사한 내용을 계획, 구현, 테스트, 수정 후 확인까지 연결하는 작업 루프를 정리해봅니다.

Week 1이 “바로 구현하지 말고 먼저 조사하게 하기”를 살펴본 글이었다면, Week 2는 그 조사 결과를 어떻게 구현, 테스트, 확인까지 연결할지에 대한 내용입니다. 즉, 이번 글의 초점은 첫 요청 문장 자체보다 작업을 끝까지 가져가는 흐름에 있습니다.

Claude Code는 빠르게 코드를 읽고 수정할 수 있습니다. 그래서 급한 버그를 만나면 바로 “고쳐줘”라고 말하고 싶어집니다. 하지만 팀에서 리뷰해야 하는 변경이라면 결과물만큼 중요한 것이 있습니다. 왜 그 파일을 바꿨는지, 어떤 테스트로 확인했는지, 추가로 확인해야 할 부분은 무엇인지입니다.

이번 글에서는 같은 버그를 두 가지 방식으로 맡겨보는 예시를 살펴봅니다. 하나는 바로 구현 요청이고, 다른 하나는 조사와 계획을 먼저 받는 요청입니다. 차이는 프롬프트 길이가 아니라, 리뷰할 때 필요한 근거를 남기느냐에 있습니다.

이번 글의 예시 버그

특정 조건에서 작업 상태 업데이트 API가 성공처럼 응답하지만, DB에는 기대한 상태가 반영되지 않는 상황을 기준으로 보겠습니다. Java 백엔드에서 자주 만나는 상태 전환, DB 업데이트 조건, 트랜잭션 범위가 얽힌 버그입니다.

1. 같은 버그, 두 가지 요청

먼저 가장 흔한 요청입니다.

일반적인 요청

작업 상태 업데이트 버그 고쳐줘.

조사부터 요청한 예시

바로 구현하지 말고 먼저 원인을 조사해줘.
관련 파일, 현재 동작, 재현 조건을 정리하고,
가장 작은 수정 계획과 테스트 방법을 제안해줘.

두 번째 요청은 Claude Code에게 먼저 근거를 만들게 합니다. 근거가 있으면 사람이 방향을 확인할 수 있고, 구현이 잘못된 방향으로 진행되는 일을 줄이는 데 도움이 됩니다.

2. 결과는 이렇게 달라집니다

같은 작업이라도 요청 방식이 달라지면 리뷰할 때 확인할 수 있는 정보가 달라집니다.

바로 구현 요청

  • 수정 속도는 빠를 수 있습니다.
  • 왜 그 파일을 바꿨는지 다시 물어봐야 할 수 있습니다.
  • 테스트 근거가 부족하면 리뷰 부담이 커집니다.

조사 후 구현 요청

  • 관련 파일과 현재 동작을 먼저 확인합니다.
  • 수정 범위와 테스트 방법이 먼저 보입니다.
  • 수정한 코드가 기존 기능에 어떤 영향을 줄지 확인하기 쉽습니다.

바로 구현 요청 후 리뷰에서 남는 질문

왜 이 파일을 바꿨는가, 다른 상태값에도 영향이 있는가, 어떤 테스트로 확인했는가를 다시 묻게 될 수 있습니다. 처음부터 이런 근거를 함께 만들게 하면 리뷰 부담을 줄이는 데 도움이 됩니다.

3. 기본 작업 루프

목표 정의
-> 코드 조사
-> 변경 계획
-> 방향 확인
-> 구현
-> 테스트
-> 수정 후 확인
-> 보완

목표 정의에서는 원하는 결과와 제약을 말합니다. 코드 조사에서는 Claude Code가 관련 파일, 호출 흐름, 기존 테스트를 찾습니다. 변경 계획에서는 수정 범위와 검증 방법을 정리합니다. 구현은 수정 범위와 테스트 방법을 한 번 확인한 뒤 진행하도록 요청할 수 있습니다.

여기서 방향 확인은 거창한 승인 절차를 말하는 것이 아닙니다. “이 파일들만 바꾸고, 이 테스트로 확인하자”는 정도를 사람이 한 번 보는 단계입니다. 이 한 단계가 있으면 Claude Code의 수정 범위가 불필요하게 넓어지거나, 테스트 방법 없이 구현부터 진행하는 일을 줄일 수 있습니다.

테스트 후에도 Claude Code에게 수정한 코드 기준으로 한 번 더 확인하게 하면 좋습니다. “무엇을 바꿨고, 어떤 동작이 영향을 받을 수 있으며, 어떤 테스트로 확인했는가”를 정리하게 하면 리뷰가 훨씬 쉬워집니다.

수정 후에 한 번 더 확인할 것

코드를 수정한 뒤에는 이번 변경이 다른 기능에 영향을 주지 않는지 확인해볼 수 있습니다. 테스트가 빠진 부분은 없는지, 필요 이상으로 많이 고친 것은 아닌지도 같이 보면 좋습니다.

4. 단계별 요청 문장 예시

매번 긴 설명을 새로 쓰지 않아도 됩니다. 단계별로 자주 쓰는 문장을 정해두면 작업 흐름을 조금 더 일정하게 가져갈 수 있습니다. 

조사:
관련 파일과 호출 흐름을 먼저 정리해줘.

계획:
가장 작은 수정안을 제안하고, 바뀔 수 있는 동작을 알려줘.

방향 확인:
내가 방향을 확인하기 전에는 파일 수정 대신,
수정 범위와 테스트 방법만 먼저 정리해줘.

구현:
계획한 범위 안에서만 수정하고,
불필요한 리팩토링은 하지 말아줘.

테스트:
어떤 테스트로 확인했는지와
실패한다면 남은 원인을 정리해줘.

확인:
수정한 코드가 다른 기능에 영향을 줄 가능성은 없는지,
테스트가 빠진 부분은 없는지 같이 봐줘.

5. 언제는 바로 맡겨도 될까

모든 작업에 긴 루프가 필요한 것은 아닙니다. 오타 수정, 주석 정리, 로그 문구 변경처럼 영향 범위가 작은 작업은 바로 맡겨도 괜찮습니다. 반대로 다음 조건 중 하나라도 있으면 먼저 조사와 계획을 받아보는 편이 안정적일 수 있습니다.

  • 수정 파일이 여러 영역에 걸칠 가능성이 있다.
  • 기존 설계 의도를 알아야 한다.
  • 트랜잭션 범위, DB 업데이트 조건, 상태 전환처럼 영향 범위가 크다.
  • 배치 처리, 외부 시스템 연동, 예외 처리, 재시도 로직에 영향이 있다.
  • 테스트가 실패했는데 원인이 명확하지 않다.
  • 리팩토링처럼 정답보다 판단이 중요한 작업이다.

참고해볼 요청 예시

버그 수정이나 테스트 실패를 맡길 때 상황에 맞게 줄이거나 바꿔서 사용할 수 있습니다.

이 작업은 바로 구현하지 말고 다음 순서로 진행해줘.
1. 관련 파일과 현재 동작을 조사해줘.
2. 원인 후보와 가장 작은 수정 계획을 제안해줘.
3. 내가 수정 범위와 테스트 방법을 확인하기 전에는 파일 수정 대신 계획만 정리해줘.
4. 확인한 범위 안에서 구현해줘.
5. 테스트 또는 확인 방법을 실행하고 결과를 정리해줘.
6. 마지막에 수정한 코드가 기존 기능에 어떤 영향을 줄 수 있는지 확인해줘.

가볍게 참고해볼 점

원인이 명확하지 않거나 영향 범위가 있는 버그 수정에서는 바로 구현을 맡기기보다 조사, 계획, 방향 확인, 구현, 테스트, 수정 후 확인 순서로 요청해볼 수 있습니다. 모든 작업에 적용할 필요는 없지만, 리뷰할 근거가 필요한 작업에서 도움이 될 수 있습니다.

한 문장으로 정리

“먼저 조사해줘”에서 한 단계 더 나아가, 조사한 내용을 구현, 테스트, 수정 후 확인까지 이어가는 방식도 함께 생각해볼 수 있습니다.
