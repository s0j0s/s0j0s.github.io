Week 2

조사, 계획, 테스트까지 이어지는 작업 루프

약 4분 · 실무 활용 팁

Week 1에서 살펴본 “먼저 조사해줘”를 바탕으로, 조사 결과를 계획, 구현, 테스트, 변경 영향 검토까지 연결하는 흐름을 정리합니다.

3줄 요약

  • “바로 고쳐줘”는 빠르지만 조사 근거와 수정 범위가 흐려지기 쉽습니다.
  • 조사, 계획, 방향 확인, 구현, 테스트, 변경 영향 검토 순서로 맡기면 리뷰할 근거가 남습니다.
  • 이번 글에서는 버그 수정을 맡길 때 이 흐름을 어떻게 활용할 수 있는지 살펴봅니다.

이번 글의 초점은 첫 요청 문장이 아니라 조사 결과를 작업 계획으로 바꾸고, 구현과 테스트를 거쳐 리뷰 가능한 근거로 남기는 과정에 있습니다.

이번 글의 예시 버그

특정 조건에서 작업 상태 업데이트 API는 성공 응답을 반환하지만, DB에는 기대한 상태가 반영되지 않는 상황을 기준으로 보겠습니다. Java 백엔드에서 자주 만나는 상태 전환, DB 업데이트 조건, 트랜잭션 범위가 얽힌 버그입니다.

1. 조사한 뒤 수정 범위를 확인하기

조사가 끝나면 바로 코드를 수정하기보다, 확인된 원인과 수정할 범위, 테스트할 내용을 먼저 정리합니다. 

조사 결과
- 업데이트된 행 수가 0건이어도 성공을 반환할 가능성이 있다.
- 이 상황을 검증하는 테스트가 없다.

수정 전에 확인할 내용
- 업데이트된 행 수가 0건이면 기존 예외 처리 방식에 따라 실패로 처리한다.
- 상태 업데이트 로직만 수정한다.
- 허용된 상태 전환과 거부된 상태 전환 테스트를 추가한다.

확인 후 요청
위 범위대로 구현하고 테스트 결과를 정리해줘.

이렇게 정리하면 사람이 수정 방향을 먼저 확인할 수 있고, 구현 후에도 계획한 범위만 바뀌었는지 같은 기준으로 검토할 수 있습니다.

2. 계획을 확인하면 달라지는 점

조사 결과를 바로 구현하는 경우와 변경 계획을 먼저 확인하는 경우를 비교해보겠습니다.

바로 구현을 맡긴 경우

  • 수정 속도는 빠를 수 있습니다.
  • 필요한 범위보다 수정 범위가 넓어질 수 있습니다.
  • 테스트 근거가 부족하면 리뷰 부담이 커집니다.

계획을 먼저 확인한 경우

  • 수정할 파일과 제외할 범위가 먼저 보입니다.
  • 구현 전에 테스트 방법을 확인할 수 있습니다.
  • 수정한 코드가 기존 기능에 어떤 영향을 줄지 확인하기 쉽습니다.

구현 후 리뷰에 남기 쉬운 질문

왜 이 파일을 바꿨는가, 다른 상태값에도 영향이 있는가, 어떤 테스트로 확인했는가를 다시 묻게 될 수 있습니다. 이 근거를 구현 전에 정리하면 리뷰할 때 다시 확인할 내용이 줄어듭니다.

3. 기본 작업 루프

목표 정의
-> 코드 조사
-> 변경 계획
-> 방향 확인
-> 구현
-> 테스트
-> 변경 영향 검토

목표 정의에서는 원하는 결과와 제약을 말합니다. 코드 조사에서는 Claude Code가 관련 파일, 호출 흐름, 기존 테스트를 찾습니다. 변경 계획에서는 수정 범위와 검증 방법을 정리합니다. 구현은 수정 범위와 테스트 방법을 한 번 확인한 뒤 진행하도록 요청할 수 있습니다.

여기서 방향 확인은 거창한 승인 절차를 말하는 것이 아닙니다. “이 파일들만 바꾸고, 이 테스트로 확인하자”는 정도를 사람이 한 번 보는 단계입니다. 이 한 단계가 있으면 Claude Code의 수정 범위가 불필요하게 넓어지거나, 테스트 방법 없이 구현부터 진행하는 일을 줄일 수 있습니다.

테스트 후에는 Claude Code에게 수정한 코드를 기준으로 영향 범위를 한 번 더 확인하게 합니다. “무엇을 바꿨고, 어떤 동작이 영향을 받을 수 있으며, 어떤 테스트로 확인했는가”를 정리하게 하면 리뷰가 훨씬 쉬워집니다.

변경 후에 한 번 더 확인할 것

코드를 수정한 뒤에는 이번 변경이 다른 기능에 영향을 주지 않는지 확인해볼 수 있습니다. 테스트가 빠진 부분은 없는지, 필요 이상으로 많이 고친 것은 아닌지도 같이 보면 좋습니다.

4. 언제는 바로 맡겨도 될까

모든 작업에 긴 루프가 필요한 것은 아닙니다. 오타 수정, 주석 정리, 로그 문구 변경처럼 영향 범위가 작은 작업은 바로 맡겨도 괜찮습니다. 반대로 다음과 같은 작업에서는 조사와 계획을 먼저 받아보는 방식도 도움이 됩니다.

  • 수정 파일이 여러 영역에 걸치거나 기존 설계 의도를 알아야 한다.
  • 트랜잭션 범위, DB 업데이트 조건, 상태 전환처럼 영향 범위가 크다.
  • 배치 처리, 외부 시스템 연동, 예외 처리, 재시도 로직에 영향이 있다.
  • 테스트 실패의 원인이 명확하지 않거나 판단이 필요한 리팩토링이다.

참고해볼 요청 예시

버그 수정이나 테스트 실패를 맡길 때 상황에 맞게 줄이거나 바꿔서 사용할 수 있습니다.

관련 파일과 현재 동작을 먼저 조사해줘.
원인 후보, 수정 범위, 테스트 방법을 정리한 뒤 잠시 멈춰줘.
방향을 확인하면 그 범위에서 구현하고 테스트해줘.
실행하지 못한 테스트와 남은 영향 범위가 있으면 함께 알려줘.

한 문장으로 정리

“먼저 조사해줘”에서 한 단계 더 나아가, 조사한 내용을 구현, 테스트, 변경 영향 검토까지 이어가면 리뷰할 근거가 남습니다.
