Claude Code Engineering Roadmap

Week 4

Skills로 반복 작업 표준화하기

자주 시키는 일은 매번 프롬프트로 설명하지 말고, Skill로 고정하면 됩니다.

이번 주 핵심

Skill은 잘 만든 프롬프트를 팀의 작업 습관으로 바꾸는 방법입니다. 일회성 설명이 아니라 재사용 가능한 작업 방식을 정의합니다.

Claude Code를 몇 주 쓰다 보면 반복되는 요청이 보입니다. “커밋 메시지 써줘”, “diff 리뷰해줘”, “릴리즈 노트 정리해줘”, “테스트 빠진 부분 찾아줘.” 매번 원하는 형식과 우선순위를 설명하는 것은 낭비입니다.

Skill은 이런 반복 작업을 표준화하는 장치입니다. 좋은 Skill은 Claude에게 언제 이 작업 방식을 써야 하는지, 어떤 순서로 판단해야 하는지, 어떤 출력 형식을 지켜야 하는지를 알려줍니다.

1. Skill이 필요한 순간

Skill은 복잡한 지식보다 반복되는 작업 방식에 잘 맞습니다. 특히 팀마다 기준이 다른 작업일수록 효과가 큽니다.

  • 커밋 메시지 작성
  • 코드 리뷰
  • 장애 원인 분석
  • 리팩토링 점검
  • 릴리즈 노트 정리
  • 테스트 보강 범위 찾기

2. 프롬프트와 Skill의 차이

프롬프트는 지금 한 번 필요한 지시입니다. Skill은 앞으로도 반복해서 쓸 작업 방식입니다. 같은 요청을 세 번 이상 설명하고 있다면 Skill 후보로 봐도 좋습니다.

프롬프트

이번 diff 리뷰해줘.
버그랑 테스트 누락 위주로 보고,
스타일 얘기는 적게 해줘.

Skill

코드 리뷰 요청이면 항상:
1. 버그와 회귀 위험을 먼저 본다.
2. 파일/라인 기준으로 짧게 쓴다.
3. 스타일 의견은 실제 유지보수 위험이 있을 때만 쓴다.

3. 좋은 Skill의 구성

Skill은 길게 쓸수록 좋은 것이 아닙니다. Claude가 행동으로 옮길 수 있는 규칙이 중요합니다.

  • 언제 이 Skill을 쓰는지 명확히 적는다.
  • 작업 순서를 짧게 정한다.
  • 출력 형식을 구체적으로 정한다.
  • 하지 말아야 할 일을 적는다.
  • 필요한 참고 파일이나 명령이 있으면 연결한다.

4. 작은 Skill이 낫다

하나의 거대한 Skill에 리뷰, 테스트, 릴리즈, 장애 분석을 모두 넣으면 Claude가 어느 기준을 우선해야 하는지 흐려집니다. Skill은 역할이 작을수록 재사용하기 쉽고, 팀에서도 합의하기 쉽습니다.

예를 들어 “PR 처리 Skill” 하나보다 “리뷰 Skill”, “릴리즈 노트 Skill”, “테스트 보강 Skill”로 나누는 편이 좋습니다. 작은 Skill은 수정하기도 쉽습니다. 팀의 리뷰 기준이 바뀌면 리뷰 Skill만 바꾸면 됩니다.

결제 버그를 공통 사례로 보면 “결제 코드 리뷰 Skill”과 “결제 테스트 보강 Skill”을 분리할 수 있습니다. 리뷰 Skill은 환불, 재시도, 중복 결제 같은 회귀 위험을 먼저 보게 하고, 테스트 보강 Skill은 누락된 성공, 실패, 경계 케이스를 기존 테스트 스타일에 맞춰 찾게 합니다.

5. 팀 공통 Skill과 개인 Skill

팀 공통 Skill에는 팀의 품질 기준이 들어갑니다. 코드 리뷰 형식, 커밋 메시지 규칙, 릴리즈 노트 기준처럼 결과물이 팀 전체에 영향을 주는 작업입니다. 개인 Skill에는 개인의 요약 방식, 학습 방식, 선호하는 탐색 루틴을 둘 수 있습니다.

팀에서 합의하면 좋은 기준

“세 번 이상 반복해서 설명한 요청”은 Skill 후보로 기록합니다. Skill은 도구 설정이 아니라 팀의 반복 작업을 정리하는 방식입니다.
