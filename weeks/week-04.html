Week 4 · 외부 맥락

MCP로 외부 시스템
연결하기

GitHub, Jira, Confluence를 연결하기 전에
무엇을 읽고 어디까지 맡길지 정합니다.

키보드 방향키 또는 아래 버튼으로 넘겨보세요.

코드만으로는 부족할 때

버그의 이유는 종종
코드 밖에 남아 있습니다.

Jira에는 재현 조건이, Confluence에는 설계 배경이,
GitHub에는 변경 논의가 남아 있습니다.

MCP가 맡는 일

도구를 많이 붙이는 것보다
정확한 맥락을 가져옵니다.

  1. 1Jira이슈 설명, 재현 조건, 기대 동작
  2. 2Confluence설계 규칙, 예외, 의사결정 기록
  3. 3GitHub관련 PR, diff, review comment

사람이 복사해 전달하던 작업 정보를 Claude Code가 출처에서 직접 확인합니다.

프로젝트 연결과 비밀 값

연결 정보는 공유해도,
인증 정보는 분리합니다.

팀과 공유 ./.mcp.json

프로젝트에서 사용할 서버와 연결 방식

회사 방식으로 관리 토큰 · 인증 정보

환경변수나 회사가 정한 보안 저장소

.mcp.json은 저장소 루트에 둡니다. 토큰 값은 저장소에 넣지 않습니다.

권한은 세 칸으로

연결하기 전에
어디까지 맡길지 나눕니다.

  1. 1읽기부터이슈, 문서, PR과 diff 조회
  2. 2확인 후 쓰기댓글 작성, PR 설명 업데이트
  3. 3별도 논의상태 변경, 삭제, 배포 관련 작업

MCP를 연결했다는 사실이 모든 작업을 허용한다는 뜻은 아닙니다.

처음 연결하는 순서

하나의 출처를
읽기 전용으로 시작합니다.

  1. Step 1자주 보는 시스템 하나 고르기
  2. Step 2필요한 저장소와 프로젝트만 허용하기
  3. Step 3/mcp로 연결과 인증 상태 확인하기
  4. Step 4조회 결과의 출처와 정확도 확인하기

연결 목록보다 무엇을 읽을 수 있고 무엇을 할 수 없는지 아는 것이 먼저입니다.

여러 출처를 함께 볼 때

연결 개수보다
읽는 순서를 정합니다.

  1. 1업무 맥락Jira에서 재현 조건과 기대 동작
  2. 2설계 기준Confluence에서 규칙과 예외
  3. 3변경 기록GitHub에서 관련 PR과 최근 diff

마지막에 코드의 호출 흐름과 테스트를 찾아 수정 계획을 제안하게 합니다.

언제 꺼내 쓸까

외부 맥락이 판단을 바꿀 때
MCP가 잘 맞습니다.

잘 맞는 작업 이슈 기반 버그 수정
PR 리뷰
릴리즈 영향 확인

문서와 코드의 불일치까지 함께 확인

덜 필요한 작업 작은 오타 수정
단일 파일 정리
명확한 테스트 추가

대화 안에 필요한 맥락이 이미 있는 작업

실전 요청

출처와 순서,
쓰기 경계를 함께 적습니다.

Jira MES-1234의 재현 조건과 기대 동작을 읽어줘.
Confluence에서 관련 상태 전환 규칙을 확인하고,
GitHub에서 관련 PR과 최근 변경을 찾아줘.

아직 코드는 수정하지 말고 원인 후보와 테스트만 정리해줘.
댓글 작성이나 상태 변경은 내가 확인하기 전에는 하지 마.

이번 주 실천

Jira 이슈 하나를 골라
조사 결과만 받아봅니다.

읽기부터, 확인 후 쓰기, 별도 논의.
맡길 일을 세 칸으로 나누면 충분합니다.

한 문장으로 정리

MCP는 연결의 수보다,
읽는 순서와
권한의 경계가 중요합니다.

다음 주에는 반복하는 작업 절차를 Skill로 묶어봅니다.
