Week 4 of 8 반복 작업

Week 4

Skill로 반복 설명 줄이기

자주 반복해서 설명하는 작업 절차는 매번 다시 쓰지 않고 Skill로 분리해둘 수 있습니다.

3줄 요약

  • 같은 설명을 여러 번 반복하고 있다면 Skill 후보로 검토해볼 수 있습니다.
  • Skill에는 반복 작업의 절차와 해당 작업에서 필요한 참고 자료를 함께 담을 수 있습니다.
  • 리뷰, 테스트 보강, 장애 원인 분석처럼 익숙한 작업부터 시험해볼 수 있습니다.

Claude Code를 몇 주 쓰다 보면 반복되는 요청이 보입니다. “커밋 메시지 써줘”, “수정 내용 리뷰해줘”, “릴리즈 노트 정리해줘”, “테스트 빠진 부분 찾아줘.” 매번 원하는 형식과 우선순위를 다시 설명하다 보면 작업 흐름이 길어질 수 있습니다.

Skill은 이런 반복 요청에서 작업 순서와 답변 형식을 미리 정해두는 장치입니다. 좋은 Skill은 언제 이 작업 방식을 쓰면 좋은지, 어떤 순서로 확인하면 되는지, 어떤 형식으로 답하면 읽기 쉬운지를 알려줍니다.

1. 반복 설명이 보일 때

프롬프트는 지금 한 번 필요한 지시입니다. Skill은 반복되는 작업 절차나 특정 작업에서 필요한 참고 지식을 다시 꺼내 쓰는 방법입니다. 커밋 메시지 작성, 코드 리뷰, 테스트 보강처럼 같은 기준을 여러 번 설명하고 있다면 Skill 후보로 생각해볼 수 있습니다.

프롬프트

이번 수정 내용을 리뷰해줘.
버그랑 테스트 누락 위주로 보고,
스타일 얘기는 적게 해줘.

Skill

코드 리뷰 요청이면 항상:
1. 버그와 회귀 위험을 먼저 본다.
2. 파일/라인 기준으로 짧게 쓴다.
3. 스타일 의견은 실제 유지보수 위험이 있을 때만 쓴다.

2. 작게 만드는 Skill

Skill은 길게 쓰기보다 역할과 확인 순서를 분명하게 적는 편이 읽기 쉽습니다. Claude Code에서는 보통 .claude/skills/스킬이름/SKILL.md 형태로 둡니다.

  • 언제 이 Skill을 쓰면 좋은지 적습니다.
  • 작업 순서를 짧게 남깁니다.
  • 읽기 편한 답변 형식을 적습니다.
  • 필요한 참고 파일이나 제외할 작업 범위를 연결합니다.

하나의 거대한 Skill에 리뷰, 테스트, 릴리즈, 장애 분석을 모두 넣으면 Claude가 어느 포인트를 우선해야 하는지 흐려집니다. 처음에는 리뷰와 테스트 보강처럼 역할을 나누면 결과를 비교하고 고치기 쉽습니다. 리뷰 Skill은 트랜잭션 범위, DB 업데이트 조건, 예외 처리 누락을 먼저 보게 하고, 테스트 보강 Skill은 성공, 실패, 경계 케이스를 기존 JUnit 스타일에 맞춰 찾게 합니다.

3. 참고할 만한 Skill 후보

관심이 있다면 팀에서 자주 반복하는 요청 하나를 골라 가볍게 시험해볼 수 있습니다.

backend-review

  • PR이나 수정 내용 리뷰 요청에 사용합니다.
  • 트랜잭션, DB 업데이트 조건, 예외 처리, 테스트 누락을 먼저 봅니다.
  • 출력은 파일/라인 기준으로 짧게 씁니다.

junit-test-planner

  • 버그 수정 전후에 필요한 JUnit 케이스를 찾습니다.
  • 성공, 실패, 경계, 재시도 케이스를 분리합니다.
  • 기존 테스트 스타일을 먼저 확인합니다.

issue-investigator

  • 프롬프트나 파일로 제공된 Jira 이슈의 원인 분석에 사용합니다.
  • 구현하지 않고 호출 흐름과 원인 후보만 정리합니다.
  • 확인할 로그, 테스트, 관련 파일을 남깁니다.

공개된 Java 코드 리뷰 Skill이나 범용 코드 리뷰 Skill을 참고할 수도 있지만, 우리 팀의 Java 8, Oracle 11g, 기존 JUnit 스타일에 맞는 기준만 골라볼 수 있습니다. 여러 사람이 반복해서 쓰는 기준은 공통 Skill로, 개인의 요약 방식이나 탐색 순서는 개인 Skill로 둘 수 있습니다.

4. 실전 Skill 예시

예를 들어 공통 리뷰 Skill을 만든다면 아래처럼 시작할 수 있습니다.

.claude/skills/backend-review/SKILL.md

---
description: >-
  Java 8 백엔드 변경에서 트랜잭션, DB 조건, 예외 처리,
  테스트 누락을 리뷰할 때 사용
---

Java 백엔드 수정 내용을 리뷰할 때는 다음 순서로 본다.

1. 트랜잭션 범위와 DB 업데이트 조건을 먼저 확인한다.
2. 예외 처리, 재시도, 중복 처리 누락을 찾는다.
3. 기존 JUnit 테스트 스타일에 맞춰 빠진 케이스를 제안한다.
4. 스타일 의견은 실제 장애나 유지보수 위험이 있을 때만 쓴다.
5. 출력은 위험도 높은 순서로 파일/라인 기준으로 짧게 쓴다.
/backend-review

현재 브랜치의 상태 업데이트 로직 변경을 리뷰해줘.

5. 가볍게 시험해보는 순서

처음부터 여러 Skill을 만들기보다 backend-review 하나를 작은 PR에 적용해볼 수 있습니다. 

1. 저장소에 .claude/skills/backend-review/SKILL.md 초안을 만든다.
2. 작은 PR 하나를 골라 /backend-review로 리뷰를 요청해본다.
3. Claude가 낸 리뷰 중 쓸모없는 의견과 빠진 내용을 기록한다.
4. 2~3번 써본 뒤 유용한 기준만 SKILL.md에 남긴다.

한 번에 공통 방식으로 정하기보다 실제 리뷰에 시험해보고, 팀에 도움이 되는 기준이 확인되면 함께 다듬어도 됩니다.

한 문장으로 정리

Skill은 자주 반복하는 요청의 순서와 답변 형식을 짧게 정리해, 매번 같은 설명을 다시 쓰지 않게 해주는 작업 단위입니다.
