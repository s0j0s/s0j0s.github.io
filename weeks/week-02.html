Claude Code Engineering Roadmap

Week 2

바로 구현시키지 않는 Claude Code 작업 루프

Claude Code는 “고쳐줘”보다 “먼저 조사하고, 계획하고, 검증해줘”라고 맡길 때 훨씬 안정적으로 일합니다.

이번 주 핵심

좋은 루프는 Claude의 속도를 살리면서도 사람이 통제권을 잃지 않게 해줍니다. 바로 구현시키는 대신 조사, 계획, 구현, 테스트, 리뷰를 작은 단계로 나눕니다.

Claude Code를 처음 팀에 들이면 가장 먼저 나오는 요청은 보통 간단합니다. “이 버그 고쳐줘”, “이 화면 바꿔줘”, “테스트 추가해줘.” 속도는 빠릅니다. 그런데 바로 구현부터 시작하면 Claude가 엉뚱한 파일을 고치거나, 기존 설계를 놓치거나, 테스트 없이 끝냈다고 말하는 일이 생깁니다.

문제는 Claude가 부족해서가 아니라, 맡긴 일의 모양이 너무 크고 흐릿하기 때문입니다. 사람에게도 “이거 알아서 잘 고쳐줘”라고만 말하면 리뷰 비용이 커집니다. Claude에게는 더더욱 작업 루프가 필요합니다.

1. 흔한 실패 패턴

바로 구현 요청은 짧아서 편하지만, 실제 개발에서는 생략된 정보가 많습니다. 무엇이 버그인지, 어디까지 바꿔도 되는지, 어떤 테스트를 통과해야 하는지, 어떤 설계 원칙을 지켜야 하는지가 빠지기 쉽습니다.

부족한 요청

로그인 버그 고쳐줘.

더 나은 요청

로그인 버그 원인을 먼저 조사해줘.
관련 파일, 재현 흐름, 의심 지점을 정리하고
수정 계획을 제안해줘.
내가 확인하면 구현까지 진행해줘.

두 요청의 차이는 길이가 아닙니다. 두 번째 요청은 Claude가 먼저 근거를 만들게 합니다. 근거가 있으면 사람이 방향을 확인할 수 있고, 구현이 잘못된 곳으로 달리는 것을 줄일 수 있습니다.

2. 권장 루프

실무에서 가장 무난한 기본 루프는 다음 순서입니다. 

목표 정의
-> 코드 조사
-> 변경 계획
-> 구현
-> 테스트
-> diff 리뷰
-> 수정

목표 정의에서는 사용자가 원하는 결과와 제약을 말합니다. 코드 조사에서는 Claude가 관련 파일, 호출 흐름, 기존 테스트를 찾습니다. 변경 계획에서는 수정 범위와 검증 방법을 정리합니다. 구현은 그 다음입니다.

테스트 후에는 끝이 아닙니다. Claude에게 diff 기준으로 리스크를 다시 보게 해야 합니다. “내가 무엇을 바꿨고, 어떤 동작이 영향을 받을 수 있으며, 어떤 테스트로 확인했는가”를 말하게 하면 리뷰가 훨씬 쉬워집니다.

3. 작은 작업과 큰 작업을 나누는 기준

모든 작업에 긴 절차가 필요한 것은 아닙니다. 오타 수정, 단순 문구 변경, 작은 테스트 케이스 추가는 바로 맡겨도 됩니다. 반대로 다음 조건 중 하나라도 있으면 먼저 조사와 계획을 받는 편이 좋습니다.

  • 수정 파일이 여러 영역에 걸칠 가능성이 있다.
  • 기존 설계 의도를 알아야 한다.
  • 데이터 마이그레이션, 인증, 결제, 권한처럼 리스크가 크다.
  • 테스트가 실패했는데 원인이 명확하지 않다.
  • 리팩토링처럼 정답보다 판단이 중요한 작업이다.

4. 실전 요청 예시

팀에서 자주 쓰는 요청은 몇 가지 형태로 고정해두면 좋습니다. 

버그 수정:
이 증상의 원인을 먼저 조사해줘. 관련 파일과 재현 경로를 정리하고,
가장 작은 수정안을 제안한 뒤 구현해줘. 마지막에 테스트 결과와 diff 리스크를 요약해줘.

리팩토링:
바로 수정하지 말고 중복 구조와 변경 위험을 먼저 분석해줘.
동작을 바꾸지 않는 범위에서 단계별 계획을 세우고, 각 단계마다 테스트 방법을 제안해줘.

테스트 추가:
현재 테스트 구조를 먼저 확인하고 누락된 케이스를 정리해줘.
기존 스타일을 따라 테스트를 추가하고, 실패할 때 원인을 분석해줘.

코드 리뷰:
diff 기준으로 버그, 회귀 위험, 누락된 테스트를 우선순위대로 리뷰해줘.
스타일 의견보다 실제 동작 리스크를 먼저 봐줘.

5. 사람이 판단해야 할 경계

Claude에게 많이 맡길수록 사람의 역할이 사라지는 것이 아니라 더 선명해집니다. Claude는 빠르게 조사하고 구현하고 테스트할 수 있습니다. 하지만 제품 의도, 허용 가능한 리스크, 배포 타이밍, 팀의 장기 설계 방향은 사람이 판단해야 합니다.

팀에서 합의하면 좋은 기준

“작업 범위가 불명확하면 구현 전에 조사와 계획을 받는다.” 이 기준 하나만 있어도 Claude Code 사용 방식이 훨씬 안정됩니다.
