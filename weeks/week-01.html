Claude Code Engineering Roadmap

Week 1

Prompt Engineering에서 Agentic Engineering까지

Claude Code를 잘 쓰려면 명령어를 많이 외우는 것보다, 일을 맡기는 방식을 바꾸는 것이 먼저입니다.

이번 주 핵심

이번 글은 개념을 깊게 파는 글이 아니라, 앞으로 8주 동안 다룰 지도를 먼저 그리는 글입니다. 핵심은 Claude Code를 “답변 도구”가 아니라 조사하고, 계획하고, 수정하고, 검증하는 작업 루프 안에 두는 것입니다.

AI 도구를 처음 쓸 때는 대부분 이렇게 시작합니다. “이 코드 고쳐줘”, “React 컴포넌트 만들어줘”, “에러 원인 알려줘”처럼 질문을 잘 쓰는 데 집중합니다. 이것이 Prompt Engineering입니다.

그런데 실제 개발은 질문 하나로 끝나지 않습니다. 코드를 고치려면 프로젝트 구조, 팀의 규칙, 테스트 방법, 배포 제약, 기존 설계 의도까지 알아야 합니다. 그래서 Claude Code를 잘 쓰는 방향은 “질문을 잘 쓰기”에서 “일을 잘 맡기기”로 바뀌고 있습니다.

예를 들어 결제 버그를 Claude Code에 맡기는 팀을 생각해보겠습니다. “결제가 실패해, 고쳐줘”라고 바로 맡기면 빠를 수는 있지만, 원인 조사와 검증 기준이 비어 있습니다. 이 시리즈는 같은 결제 버그를 조사, 계획, 구현, 리뷰, 안전장치까지 이어지는 흐름으로 다룹니다.

1. Prompt Engineering: 질문을 잘 쓰기

Prompt Engineering은 AI에게 원하는 답을 얻기 위해 질문을 잘 작성하는 방법입니다. Claude Code를 처음 쓸 때 가장 익숙한 방식입니다. 

React로 todo app 만들어줘.
이 코드를 리팩토링해줘.
이 에러를 고쳐줘.

이 방식은 빠르지만 개발 업무 전체를 맡기기에는 부족합니다. 결제 버그처럼 원인, 영향 범위, 테스트 기준이 중요한 작업은 질문 하나보다 작업 루프가 필요합니다.

부족한 요청

이 버그 고쳐줘.

더 나은 요청

이 버그의 원인을 먼저 찾아줘.
관련 파일을 정리하고,
수정 계획을 제안한 뒤,
내가 확인하면 구현해줘.

2. Context Engineering: 필요한 배경을 알려주기

Context Engineering은 AI가 무슨 정보를 보고 판단할지 설계하는 일입니다. 쉽게 말하면 Claude에게 “지금 이 작업을 하려면 이 배경을 먼저 봐라”라고 정리해주는 것입니다. 자세한 방법은 Week 3의 CLAUDE.md에서 다룹니다.

여기서는 프로젝트 구조, 도메인 규칙, 테스트 명령, 금지된 리팩토링 범위 같은 배경이 들어갑니다. 결제 버그라면 “결제 도메인 규칙은 어디에 있는가”, “어떤 테스트를 돌려야 하는가”가 먼저 Claude에게 보여야 합니다.

3. Harness Engineering: 일할 수 있는 도구를 연결하기

Harness Engineering은 AI가 어떻게 일할지 설계하는 일입니다. 같은 Claude라도 어떤 도구를 쓸 수 있는지, 어떤 순서로 움직이는지, 결과를 어떻게 검증하는지에 따라 완전히 다른 작업자가 됩니다. 테스트, 권한, 경고, 차단 같은 안전장치는 Week 7의 Hooks에서 더 구체적으로 다룹니다.

여기에는 파일 접근, 터미널 명령, 테스트 실행, 권한 승인, 외부 시스템 연결이 포함됩니다. 결제 버그 조사 중 GitHub Issue를 읽고, 관련 테스트를 실행하고, 위험한 쓰기 작업은 승인 후 진행하게 만드는 장치입니다. 

이 버그의 원인을 먼저 조사해줘.
관련 파일과 호출 흐름을 정리한 뒤 수정 계획을 제안해줘.
구현 후 테스트를 실행하고 실패하면 원인을 다시 분석해줘.
마지막으로 diff 기준으로 리스크를 설명해줘.

이런 요청을 하면 Claude는 단순히 답만 하는 것이 아니라 작업 과정에 들어옵니다. 중요한 것은 “코드를 빨리 생성하는 능력”이 아니라 “근거를 찾고, 변경하고, 검증하는 흐름”입니다.

간단히 말하면 Context는 무엇을 보여줄지, Harness는 어떻게 일하게 할지입니다. Context가 없으면 AI가 엉뚱한 정보를 바탕으로 판단하고, Harness가 없으면 맞는 판단을 해도 실제 변경과 검증으로 이어지지 않을 수 있습니다.

4. Agentic Engineering: 작은 작업 루프로 맡기기

Agentic Engineering은 AI가 모든 것을 알아서 한다는 뜻이 아닙니다. 사람이 목표와 기준을 정하고, Claude가 조사, 구현, 테스트, 요약 같은 작업을 반복해서 수행하는 방식입니다.

핵심은 한 번에 크게 시키지 않고, 다음 루프를 반복하는 것입니다.

목표 정의
-> 코드 조사
-> 변경 계획
-> 구현
-> 테스트
-> 리뷰
-> 수정

Claude Code를 효율적으로 쓰는 팀은 이 루프를 명확하게 만듭니다. 반대로 Claude Code를 “코드 빨리 짜주는 도구”로만 쓰면 품질이 흔들릴 수 있습니다. 변경 의도를 놓치거나, 테스트가 부족하거나, 기존 설계와 맞지 않는 코드가 들어갈 수 있습니다.

5. 그래서 팀의 사용법이 중요합니다

Claude Code를 개인이 각자 다르게 쓰면 생산성은 일시적으로 오를 수 있습니다. 하지만 팀 전체의 개발 방식은 크게 바뀌지 않습니다. 팀 단위로 효과를 내려면 공통 사용법이 필요합니다.

  • 작업 전에는 계획을 먼저 받는다.
  • 큰 작업은 조사, 구현, 리뷰로 나눈다.
  • CLAUDE.md에 팀 규칙을 축적한다.
  • 반복 작업은 Skills로 표준화한다.
  • 외부 시스템 연결은 MCP로 관리한다.
  • 큰 작업은 SubAgent로 역할을 분리한다.
  • Hooks로 테스트, 포맷팅, 보안 검사를 자동화한다.
  • 최종 판단은 항상 사람이 diff와 테스트 결과를 보고 한다.

이 공통 사용법이 쌓이면 Claude Code는 개인 생산성 도구를 넘어 팀의 Engineering System이 됩니다.

한 문장으로 정리

Claude Code를 잘 쓰는 팀은 “좋은 프롬프트”보다 “좋은 작업 루프와 팀 규칙”을 먼저 만듭니다.

실전에서 바로 써먹는 패턴

Claude Code에게 바로 코드를 맡기기 전에, 작업의 기준을 먼저 말해주는 습관을 들이면 결과가 훨씬 안정적입니다. 요청은 짧아도 되지만, 판단 기준은 빠지면 안 됩니다. 

이 작업은 바로 구현하지 말고 먼저 조사해줘.
관련 파일과 현재 동작을 정리하고,
수정 방향을 2가지 이하로 비교한 뒤,
테스트로 확인할 수 있는 기준까지 제안해줘.

팀에서는 “바로 수정 요청”을 기본값으로 두지 않는 것만으로도 리뷰 비용이 줄어듭니다. 다음 주에는 결제 버그 같은 실무 사례를 기준으로 이 감각을 실제 작업 루프로 정리합니다.

다음 주 예고

다음 주에는 Claude Code를 더 잘 쓰기 위한 기본 작업 루프를 다룹니다. 핵심은 잘 물어보기가 아니라 일을 잘 맡기기입니다. 바로 구현시키는 방식에서 벗어나, 계획을 받고, 변경 범위를 좁히고, 테스트로 닫는 실전 패턴을 정리합니다.
