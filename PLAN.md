# Claude Code Newsletter Plan

## 목표

이 뉴스레터는 Claude Code를 이미 사용하고 있는 팀원이 쉬는 시간에 가볍게 읽고,
필요한 내용을 다음 작업에서 골라 활용할 수 있도록 돕는 8주 구성입니다.
빠른 코드 생성 자체를 강조하기보다 조사, 작은 변경, 테스트 근거,
리뷰 가능성을 중심으로 실무에서 참고할 만한 작업 패턴을 소개합니다.

`Agentic Engineering`은 AI 에이전트가 구현을 돕고 사람이 작업 범위와 품질을
책임지는 접근을 설명하는 용어로 사용합니다. 용어 자체보다 실제 요청 문장과
확인 방법을 이해하기 쉽게 전달하는 데 초점을 둡니다.

## 발행과 편집 기준

- 공개본은 데스크톱 발표에 맞춘 슬라이드 형식으로 구성하고, 발행 전 초안은
  내용을 다듬기 위한 글 형식으로 유지할 수 있습니다.
- 한 편은 약 3~6분 안에 살펴볼 수 있는 분량을 목표로 합니다.
- 슬라이드는 문제 상황, 작업 흐름, 요청 예시, 한 문장 정리가 자연스럽게
  이어지도록 구성합니다. 모든 편에 같은 순서를 억지로 적용하지는 않습니다.
- 독자에게 특정 방식을 강요하기보다 상황에 맞는 항목을 골라볼 수 있게 안내합니다.
- 전문 용어는 처음 등장할 때 짧게 설명하고 주차 사이에서 표기를 통일합니다.
- 회사 예시는 Claude Code를 AWS Bedrock 환경에서 사용하는 상황을 기준으로 합니다.
- 공개 전에는 1366×768과 1920×1080 화면에서 줄바꿈, 카드 간격, 코드와 요청 예시의
  넘침을 확인합니다. 조사, 계획, 구현처럼 서로 다른 단계는 시각적으로 구분합니다.
- 홈에서는 현재 공개하는 글만 `이번 주`로 표시하고, 이후 글은 `Coming soon`으로 둡니다.
- 현재 공개 대상은 1~2주차이며 3~8주차는 발행 전 초안입니다.

## 현재 진행 상태

- Week 1: 슬라이드 공개 완료
- Week 2: 슬라이드 공개 완료, 홈의 `이번 주`로 표시
- Week 3: 슬라이드 형식의 발행 전 초안, 홈에서는 링크하지 않음
- Week 4~8: 글 형식의 발행 전 초안, 홈에서는 링크하지 않음

## Week 1. Claude Code에게 일을 잘 맡기는 법

바로 구현을 요청하는 방식과 조사부터 요청하는 방식을 비교합니다. Prompt Engineering과
Agentic Engineering의 차이를 간단히 짚고, 관련 파일, 원인, 수정 방향, 테스트 방법을
함께 확인할 수 있는 요청 예시를 소개합니다.

## Week 2. 바로 구현하기 전에, 완료 기준부터 맞추기

관리자 주문 상태 변경 기능을 예시로, 어떤 순서로 상태를 바꿀지, 누가 바꿀 수 있는지,
같은 요청이 여러 번 오거나 저장에 실패하면 어떻게 할지 살펴봅니다. 구현 전에
`PLAN.md`로 화면, 규칙, 완료 기준을 정리하고 사람이 계획을 확인한 뒤 작업을 작은
Phase로 나눕니다. 각 Phase의 구현과 테스트 결과를 사람이 확인한 뒤 다음 Phase로
넘어가는 흐름을 다룹니다.

## Week 3. 우리 팀의 Claude Code 설정, 무엇부터 쓸까

1주차의 작업 순서와 2주차의 완료 기준을 매번 다시 설명하는 문제에서 시작합니다.
루트 `CLAUDE.md`와 `.claude/CLAUDE.md`를 함께 쓰는 팀의 구분 기준을 살펴보고,
두 파일 모두 프로젝트 업무 매뉴얼이라는 점을 짚습니다. `rules/`는 상황별 세부 지침,
`settings.json`은 실제 권한과 동작을 정하는 설정으로 구분합니다.
팀 공용 설정과 개인 설정, AWS Bedrock 환경에서 공유하면 안 되는 값도 함께 살펴봅니다.
MCP, Skill, SubAgent, Hooks는 이후 주차에서 다룰 도구 상자로만 소개합니다.

## Week 4. MCP로 외부 시스템 연결하기

GitHub, Jira, Confluence처럼 코드 밖에 있는 작업 맥락을 Claude Code와 연결하는 방법을
살펴봅니다. 저장소 루트의 `.mcp.json`과 개인·프로젝트 연결 범위를 구분하고,
연결 기능보다 정보 출처를 읽는 순서, 읽기·쓰기 권한의 범위,
쓰기 작업 전 확인이 필요한 지점을 중심으로 설명합니다.

## Week 5. Skill로 반복 설명 줄이기

매번 다시 설명하는 작업 순서와 출력 형식을 작은 Skill로 묶는 방법을 소개합니다.
Skill을 자주 하는 작업의 레시피로 이해하고 `SKILL.md`와 참고 자료가 맡는 역할을 살펴봅니다.
리뷰, 테스트 계획, 이슈 조사 예시를 통해 처음부터 크게 표준화하기보다
반복이 분명한 절차 하나부터 가볍게 시험해보는 흐름을 다룹니다.

## Week 6. SubAgent로 큰 작업 나누기

조사, 구현, 리뷰처럼 성격이 다른 일을 SubAgent 역할로 나누는 방법을 다룹니다.
작은 수정에는 기본 작업 루프를 유지하고, 원인 분석이나 독립적인 검토가 필요한 경우에만
전문 역할을 나눕니다. 독립된 맥락과 결과 전달 방식, 필요할 때만 사용하는 프로젝트 메모를
살펴보고 남용 시 주의점도 함께 소개합니다.

## Week 7. Hooks로 잊기 쉬운 확인 알려주기

테스트 확인처럼 놓치기 쉬운 항목을 Hook으로 알려주는 방법을 소개합니다.
처음부터 자동 실행이나 차단을 적용하기보다 알림부터 시작하고,
Hook 파일과 `settings.json` 등록의 관계를 살펴봅니다. 팀에서 사용해본 뒤
자동화할 항목과 사람이 판단할 항목을 나누는 흐름을 다룹니다.

## Week 8. 팀에서 편하게 꺼내 쓰는 Claude Code 메모

앞선 7주에서 사용한 요청 패턴, `.claude` 구성, Skill, MCP, SubAgent, Hooks를
“언제 무엇을 고를까”라는 선택 기준으로 먼저 정리한 뒤 작은 팀 메모로 남깁니다.
완성된 플레이북을 한 번에 만들기보다 자주 반복되는 작업 하나의 요청 방식과
확인 포인트부터 기록하고 가볍게 개선하는 방법으로 마무리합니다.
