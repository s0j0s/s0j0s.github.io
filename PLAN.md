# Claude Code Newsletter Plan

## 방향

이 뉴스레터는 Claude Code를 이미 쓰기 시작한 팀원이 쉬는 시간에 가볍게 읽고, 다음 작업에서 바로 적용할 수 있게 만드는 8주 구성이다. 목표는 AI 코딩 자체를 과장하는 것이 아니라, 회사에 도움이 되는 방향으로 Claude Code 사용 습관을 맞추는 것이다. 핵심 기준은 빠른 코드 생성보다 조사, 작은 변경, 테스트 근거, 리뷰 가능성이다.

## Week 1. Claude Code에게 일을 잘 맡기는 법

“고쳐줘”에서 한 걸음 나아가 “먼저 조사하고, 작게 고치고, 테스트 근거를 남겨줘”라고 맡기는 기본 습관을 다룬다. 팀원이 Claude Code를 켤 때 바로 바꿔볼 수 있는 요청 방식에 집중한다.

## Week 2. 바로 구현시키지 않는 Claude Code 작업 루프

조사, 계획, 승인, 구현, 테스트, diff 리뷰로 이어지는 기본 작업 루프를 정리한다. 결제 버그 같은 실무 예시를 기준으로, 바로 구현시키지 않는 이유와 리뷰 비용을 줄이는 방법을 설명한다.

## Week 3. CLAUDE.md와 Context Engineering

팀 규칙, 테스트 명령, 프로젝트 구조, 도메인 지식, 금지된 리팩토링 범위를 `CLAUDE.md`에 정리하는 방법을 다룬다. 매번 같은 설명을 반복하지 않고 Claude Code가 팀의 공통 맥락을 참고하게 만드는 것이 목표다.

## Week 4. Skills로 반복 작업 표준화하기

자주 반복되는 리뷰, 커밋 메시지 작성, 리팩토링 점검, 장애 분석 요청을 Skill로 표준화하는 방법을 다룬다. 개인별 프롬프트 팁을 팀에서 재사용 가능한 작업 방식으로 바꾸는 주차다.

## Week 5. MCP로 외부 시스템 연결하기

GitHub, Jira, 문서, API 스펙처럼 코드 밖에 있는 작업 맥락을 Claude Code와 연결하는 방법을 소개한다. 연결 자체보다 어떤 출처를 신뢰하고 어떤 권한을 줄지 정하는 기준에 초점을 둔다.

## Week 6. SubAgent로 큰 작업 나누기

큰 작업을 한 번에 맡기지 않고 조사, 구현, 리뷰 같은 역할로 나누는 방식을 다룬다. context를 아끼고 책임을 분리해 결과물을 더 검토 가능하게 만드는 것이 핵심이다.

## Week 7. Hooks로 안전장치 만들기

테스트 실행, 포맷팅, secret 검사, 위험 명령 차단처럼 Claude Code 작업 전후에 자동으로 확인할 수 있는 안전장치를 다룬다. 팀에서 자동화할 것과 사람이 판단할 것을 나누는 기준을 정리한다.

## Week 8. 팀 플레이북: Claude Code를 팀의 개발 시스템으로 만들기

지난 7주 내용을 팀 운영 규칙으로 묶는다. 팀용 slash command, 공통 요청 패턴, 리뷰 기준, 성공 기준을 정리해 Claude Code를 개인 도구가 아니라 팀의 개발 시스템으로 만드는 마무리 주차다.
