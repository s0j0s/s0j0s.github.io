Claude Code Engineering Roadmap
Week 6 of 8 역할 분리

Week 6

SubAgent로 큰 작업 나누기

큰 작업은 한 Claude에게 한 번에 맡기기보다 조사, 구현, 리뷰 역할을 나누면 품질과 맥락 관리가 좋아집니다.

3줄 요약

  • SubAgent는 큰 작업을 조사, 구현, 리뷰처럼 작은 역할로 나누는 방식입니다.
  • 역할을 나누면 한 대화 안에서 원인 분석과 수정과 리뷰가 섞이는 일을 줄일 수 있습니다.
  • 작은 수정에는 과하고, 원인 분석이 길거나 변경 범위가 넓을 때 효과가 큽니다.

이번 주 포인트

SubAgent는 Claude를 더 똑똑하게 만드는 기능이 아니라, 큰 일을 더 작고 검증 가능한 단위로 나누는 방식입니다.

리팩토링 범위가 넓거나 버그 원인이 길게 이어질 때, 한 대화에서 모든 것을 처리하려고 하면 맥락이 쉽게 흐려집니다. 처음에는 원인 분석을 하다가, 중간에는 구현을 하고, 마지막에는 리뷰까지 이어집니다. 역할이 섞이면 놓치는 부분도 생깁니다.

SubAgent는 이런 작업을 역할별로 나눌 때 유용합니다. 한쪽은 조사만 하고, 다른 쪽은 제한된 범위에서 구현하고, 또 다른 쪽은 diff를 기준으로 위험을 봅니다.

1. 필요한 상황

SubAgent를 써볼 만한 작업

  • 변경 범위가 넓은 리팩토링
  • 원인 후보가 여러 개인 버그
  • 리뷰와 구현을 분리하고 싶은 작업
  • 여러 후보 접근을 비교해야 하는 작업

쓰지 않아도 되는 작업

  • 오타 수정
  • 단일 테스트 추가
  • 작은 조건문 수정
  • 이미 원인이 명확한 버그

작은 수정까지 SubAgent로 나누면 오히려 느려집니다. SubAgent는 큰 작업의 맥락 부담을 줄이는 도구이지, 모든 일을 절차화하기 위한 도구는 아닙니다.

2. 역할 분리 예시

Investigator

  • 작업 상태 업데이트 실패 원인과 호출 흐름만 조사한다.
  • 관련 파일과 원인 후보를 정리한다.
  • 구현은 하지 않는다.

Builder

  • 확인된 작은 수정만 구현한다.
  • 기존 패턴을 따른다.
  • 테스트를 실행하고 결과를 남긴다.

Reviewer 역할도 중요합니다. Reviewer는 구현자의 설명을 믿는 것이 아니라 diff를 기준으로 봅니다. 트랜잭션 범위, DB 업데이트 조건, 누락된 테스트, 과한 변경을 우선 확인합니다.

3. 우리 팀에서 써볼 만한 SubAgent 구성

처음에는 역할을 세 개 정도로만 나누는 것이 좋습니다. 이름이 많아지면 오히려 어떤 역할을 써야 할지 고민이 늘어납니다.

Issue Investigator

  • Jira, Confluence, GitHub, 코드를 읽고 원인 후보를 좁힙니다.
  • 파일 수정은 하지 않습니다.
  • 결과로 호출 흐름, 관련 파일, 원인 후보, 확인할 테스트를 남깁니다.

Small Fix Builder

  • 사람이 확인한 수정안 하나만 구현합니다.
  • 공통 모듈 리팩토링이나 넓은 구조 변경은 하지 않습니다.
  • 수정 후 관련 JUnit 테스트 결과를 남깁니다.

Diff Reviewer

  • 구현 설명이 아니라 실제 diff를 기준으로 봅니다.
  • 회귀 위험, 테스트 누락, 과한 변경을 우선 확인합니다.
  • 문제 없으면 남은 리스크와 확인한 테스트만 짧게 남깁니다.

Test Planner

  • 기존 테스트 스타일을 먼저 확인합니다.
  • 성공, 실패, 경계, 재시도 케이스를 제안합니다.
  • 테스트 코드는 방향을 확인한 뒤 추가합니다.

써볼 만한 흐름은 Issue Investigator로 조사하고, 사람이 수정 방향을 고른 뒤 Small Fix Builder에게 구현을 맡기고, 마지막에 Diff Reviewer로 검토하는 방식입니다. 테스트가 애매한 작업만 Test Planner를 추가합니다.

4. 실제 SubAgent 파일 예시

프로젝트에서 함께 써볼 SubAgent는 .claude/agents/ 아래에 둘 수 있습니다. 먼저 파일을 수정하지 않는 조사 agent부터 만들어보면 부담이 적습니다. 

# .claude/agents/issue-investigator.md
---
name: issue-investigator
description: Use for Jira-based Java backend bug investigation before code changes
tools: Read, Grep, Glob
model: sonnet
---

You investigate Java backend bugs before implementation.

Rules:
- Do not edit files.
- Start from Jira, Confluence, or GitHub context if provided.
- Find related service, repository, adapter, and test files.
- Summarize call flow, likely causes, and verification tests.
- Output: related files, call flow, top 2 causes, test plan.
issue-investigator agent를 사용해서 MES-1234 원인을 조사해줘.
파일 수정은 하지 말고 관련 코드 위치, 호출 흐름,
원인 후보 2개, 확인할 JUnit 테스트만 정리해줘.

구현 agent를 만들 때는 권한을 더 신중하게 보는 편이 좋습니다. 처음에는 조사 agent만 팀에 공유하고, 구현은 기본 Claude Code 세션에서 사람이 방향을 확인한 뒤 진행해도 충분합니다.

5. 좋은 요청 방식

SubAgent를 쓸 때는 역할과 기대하는 결과를 짧게 적어두면 좋습니다. “조사해줘”보다 “구현하지 말고 관련 파일, 호출 흐름, 원인 후보, 검증 방법만 정리해줘”가 낫습니다.

그대로 써봐도 되는 요청

상태 업데이트 버그를 역할별로 나눠서 볼 때 사용할 수 있습니다.

Investigator:
작업 상태 업데이트 이슈의 원인을 조사해줘.
파일 수정은 하지 말고 관련 코드 위치, 호출 흐름,
가능성 높은 원인 2개, 확인해야 할 테스트만 정리해줘.

Builder:
Investigator 결과를 바탕으로 가장 작은 수정만 적용해줘.
공통 트랜잭션 처리나 외부 연동 모듈은 계획 없이 리팩토링하지 마.
구현 후 관련 JUnit 테스트를 실행해줘.

Reviewer:
이번 diff를 리뷰해줘.
버그, 회귀 위험, 테스트 누락만 우선순위대로 말해줘.

6. 남용하면 생기는 문제

SubAgent가 많아지면 각자의 결과를 통합하는 비용이 생깁니다. 작은 작업에 역할을 나누면 실제 수정 시간보다 조율 시간이 더 커질 수 있습니다. 또한 역할을 잘못 나누면 조사 결과와 구현 결과가 어긋날 수 있습니다.

그래서 마지막 판단은 사람이 하는 편이 좋습니다. 어떤 조사 결과를 채택할지, 어느 구현안을 진행할지, 리뷰 지적 중 무엇을 반영할지는 사람이 마지막에 고르면 됩니다.

팀에서 가볍게 맞춰보면 좋은 것

SubAgent는 “파일 수가 많다”보다 “역할을 나누면 더 편하겠다”는 느낌이 있을 때 써봅니다. 작은 작업은 기본 루프로 충분합니다.

이번 주 실천

최근 이슈 하나를 골라 구현 요청 전에 Investigator 역할로만 맡겨봅니다. 결과가 충분히 명확할 때만 Builder에게 작은 수정 범위를 맡겨봅니다.

이번 주에 가볍게 남겨볼 것

SubAgent를 써볼 만한 경우를 짧게 메모합니다. “작은 수정은 기본 루프”, “원인 분석이 긴 버그는 Investigator”, “확인한 수정만 Builder”, “구현 후 diff는 Reviewer”처럼 역할별 힌트를 남깁니다.
