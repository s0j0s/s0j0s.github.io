개인 팁에서 팀의 선택 기준으로

좋았던 사용법도
기록하지 않으면 다시 찾기 어렵습니다.

완성된 플레이북보다 실제로 써본 요청과 확인 포인트부터 남깁니다.

지난 7주를 세 묶음으로

팀 기준, 작업 도구,
검증 루프로 이어집니다.

  1. 1맡기는 순서조사, 계획, 방향 확인, 구현, 테스트
  2. 2설정과 도구CLAUDE.md, rules, MCP, Skill, SubAgent
  3. 3확인과 근거settings, Hooks, diff 리뷰, 테스트 결과

도구의 이름보다 지금 겪는 불편을 먼저 봅니다.

항상 참고할 내용

프로젝트 기준과 실행 범위는
설정 파일에 남깁니다.

  1. 1CLAUDE.md모든 작업에서 필요한 프로젝트 설명
  2. 2.claude/rules/특정 경로에서만 필요한 세부 지침
  3. 3.claude/settings.json허용, 확인, 차단할 행동

사실과 기준은 지침에, 실제 허용 범위는 settings에 둡니다.

필요할 때 꺼내는 도구

불편의 종류에 맞는
도구 하나를 고릅니다.

  1. 1외부 맥락 · MCPGitHub·Jira·Confluence 정보가 필요할 때
  2. 2반복 절차 · Skill같은 절차를 현재 작업에서 반복할 때
  3. 3독립 작업 · SubAgent긴 조사나 분석을 별도 맥락에 맡길 때
  4. 4자동 실행 · Hooks특정 시점에 반드시 실행하거나 차단할 때

팀 메모에 필요한 것

기능 설명보다
선택과 확인을 기록합니다.

  • 어떤 작업에서 꺼내 쓰는가
  • 어디까지 읽고 실행할 수 있는가
  • 사람이 확인할 시점은 언제인가
  • 어떤 결과를 남겨야 완료인가

담당자와 마지막 검토일도 함께 남겨 오래된 기준을 정리합니다.

  • 위험한 작업에서 사람이 한 번 더 확인하는가
