Claude Code Engineering Roadmap
Week 4 of 8 반복 작업

Week 4

Skills로 반복 작업 표준화하기

자주 시키는 일은 매번 프롬프트로 설명하지 말고, Skill로 고정하면 됩니다.

3줄 요약

  • 세 번 이상 반복해서 설명한 요청은 Skill 후보입니다.
  • Skill은 긴 프롬프트가 아니라 팀의 반복 작업 순서와 출력 형식을 정하는 장치입니다.
  • 우리 팀에서는 리뷰, 테스트 보강, 장애 원인 분석부터 작게 시작하기 좋습니다.

이번 주 핵심

Skill은 잘 만든 프롬프트를 팀의 작업 습관으로 바꾸는 방법입니다. 일회성 설명이 아니라 재사용 가능한 작업 방식을 정의합니다.

Claude Code를 몇 주 쓰다 보면 반복되는 요청이 보입니다. “커밋 메시지 써줘”, “diff 리뷰해줘”, “릴리즈 노트 정리해줘”, “테스트 빠진 부분 찾아줘.” 매번 원하는 형식과 우선순위를 설명하는 것은 낭비입니다.

Skill은 이런 반복 작업을 표준화하는 장치입니다. 좋은 Skill은 Claude에게 언제 이 작업 방식을 써야 하는지, 어떤 순서로 판단해야 하는지, 어떤 출력 형식을 지켜야 하는지를 알려줍니다.

1. Skill이 필요한 순간

Skill은 복잡한 지식보다 반복되는 작업 방식에 잘 맞습니다. 특히 팀마다 기준이 다른 작업일수록 효과가 큽니다.

  • 커밋 메시지 작성
  • 코드 리뷰
  • 장애 원인 분석
  • 리팩토링 점검
  • 릴리즈 노트 정리
  • 테스트 보강 범위 찾기

2. 프롬프트와 Skill의 차이

프롬프트는 지금 한 번 필요한 지시입니다. Skill은 앞으로도 반복해서 쓸 작업 방식입니다. 같은 요청을 세 번 이상 설명하고 있다면 Skill 후보로 봐도 좋습니다.

프롬프트

이번 diff 리뷰해줘.
버그랑 테스트 누락 위주로 보고,
스타일 얘기는 적게 해줘.

Skill

코드 리뷰 요청이면 항상:
1. 버그와 회귀 위험을 먼저 본다.
2. 파일/라인 기준으로 짧게 쓴다.
3. 스타일 의견은 실제 유지보수 위험이 있을 때만 쓴다.

3. 좋은 Skill의 구성

Skill은 길게 쓸수록 좋은 것이 아닙니다. Claude가 행동으로 옮길 수 있는 규칙이 중요합니다.

  • 언제 이 Skill을 쓰는지 명확히 적는다.
  • 작업 순서를 짧게 정한다.
  • 출력 형식을 구체적으로 정한다.
  • 하지 말아야 할 일을 적는다.
  • 필요한 참고 파일이나 명령이 있으면 연결한다.

4. 작은 Skill이 낫다

하나의 거대한 Skill에 리뷰, 테스트, 릴리즈, 장애 분석을 모두 넣으면 Claude가 어느 기준을 우선해야 하는지 흐려집니다. Skill은 역할이 작을수록 재사용하기 쉽고, 팀에서도 합의하기 쉽습니다.

예를 들어 “PR 처리 Skill” 하나보다 “리뷰 Skill”, “릴리즈 노트 Skill”, “테스트 보강 Skill”로 나누는 편이 좋습니다. 작은 Skill은 수정하기도 쉽습니다. 팀의 리뷰 기준이 바뀌면 리뷰 Skill만 바꾸면 됩니다.

Java 백엔드 작업을 공통 사례로 보면 “상태 전환 리뷰 Skill”과 “테스트 보강 Skill”을 분리할 수 있습니다. 리뷰 Skill은 트랜잭션 범위, DB 업데이트 조건, 예외 처리 누락을 먼저 보게 하고, 테스트 보강 Skill은 성공, 실패, 경계 케이스를 기존 JUnit 스타일에 맞춰 찾게 합니다.

5. 우리 팀에서 시작하기 좋은 Skills Set

처음부터 많은 Skill을 만들 필요는 없습니다. 자주 반복되고, 출력 기준을 맞추면 바로 도움이 되는 작업부터 시작하는 편이 좋습니다.

backend-review

  • PR이나 diff 리뷰 요청에 사용합니다.
  • 트랜잭션, DB 업데이트 조건, 예외 처리, 테스트 누락을 먼저 봅니다.
  • 출력은 파일/라인 기준으로 짧게 씁니다.

junit-test-planner

  • 버그 수정 전후에 필요한 JUnit 케이스를 찾습니다.
  • 성공, 실패, 경계, 재시도 케이스를 분리합니다.
  • 기존 테스트 스타일을 먼저 확인합니다.

issue-investigator

  • Jira 이슈 기반 원인 분석에 사용합니다.
  • 구현하지 않고 호출 흐름과 원인 후보만 정리합니다.
  • 확인할 로그, 테스트, 관련 파일을 남깁니다.

release-note-writer

  • 배포 전 변경 내용을 요약할 때 사용합니다.
  • 사용자 영향, 운영 영향, 확인한 테스트를 분리합니다.
  • 내부 구현 세부사항은 줄입니다.

java8-compat-review

  • Java 9 이상 문법이나 API가 들어왔는지 확인합니다.
  • List.of, Map.of, var, Stream.toList 같은 사용을 찾습니다.
  • Java 8에서 가능한 대안을 같이 제안합니다.

transaction-review

  • 상태 변경과 DB 업데이트가 있는 작업에 사용합니다.
  • rollback 조건, 중복 처리, partial update 위험을 확인합니다.
  • 외부 연동 전후의 실패 케이스를 따로 봅니다.

oracle11g-sql-review

  • SQL, MyBatis XML, JDBC query 문자열을 리뷰할 때 사용합니다.
  • LIMIT, FETCH FIRST, OFFSET 같은 11g 미지원 문법을 찾습니다.
  • ROWNUM, ROW_NUMBER(), sequence 기반 대안을 제안합니다.

oracle11g-pagination-review

  • 목록 조회와 페이징 쿼리를 만들 때 사용합니다.
  • ORDER BY와 ROWNUM 적용 순서를 확인합니다.
  • 결과 순서가 흔들리지 않도록 정렬 기준을 같이 봅니다.

6. 팀 공통 Skill과 개인 Skill

팀 공통 Skill에는 팀의 품질 기준이 들어갑니다. 코드 리뷰 형식, 커밋 메시지 규칙, 릴리즈 노트 기준처럼 결과물이 팀 전체에 영향을 주는 작업입니다. 개인 Skill에는 개인의 요약 방식, 학습 방식, 선호하는 탐색 루틴을 둘 수 있습니다.

7. 팀에 적용하는 순서

처음부터 모든 Skill을 만들기보다 하나를 실제 PR에 써보고 다듬는 것이 좋습니다. 아래 순서로 시작하면 부담이 적습니다. 

1. 저장소에 .claude/skills/backend-review/SKILL.md 파일을 만든다.
2. 작은 PR 하나를 골라 /backend-review로 리뷰를 요청한다.
3. Claude가 낸 리뷰 중 쓸모없는 의견과 빠진 기준을 기록한다.
4. SKILL.md에서 출력 형식과 확인 순서를 조정한다.
5. 2~3번 반복해서 팀 기준에 맞으면 저장소에 커밋한다.
6. 그 다음 java8-compat-review, junit-test-planner를 하나씩 추가한다.
7. SQL 변경이 많은 저장소라면 oracle11g-sql-review를 추가한다.
8. MyBatis XML이나 페이징 쿼리가 많다면 oracle11g-pagination-review를 추가한다.

팀 공통 Skill은 한 번에 완성하려고 하지 않아도 됩니다. “우리 팀 리뷰에서 자주 놓치는 것”을 하나씩 반영하면 시간이 지날수록 실제 기준에 가까워집니다.

팀에서 합의하면 좋은 기준

“세 번 이상 반복해서 설명한 요청”은 Skill 후보로 기록합니다. Skill은 도구 설정이 아니라 팀의 반복 작업을 정리하는 방식입니다.

실전 Skill 예시

아래 파일을 만들면 프로젝트 안에서 /backend-review처럼 사용할 수 있습니다.

# .claude/skills/backend-review/SKILL.md
---
description: Java 8 backend diff review for transaction, DB update, exception handling, and missing tests
allowed-tools: Read Grep Glob
---

Java backend diff를 리뷰할 때는 다음 순서로 본다.

1. 트랜잭션 범위와 DB 업데이트 조건을 먼저 확인한다.
2. 예외 처리, 재시도, 중복 처리 누락을 찾는다.
3. 기존 JUnit 테스트 스타일에 맞춰 빠진 케이스를 제안한다.
4. 스타일 의견은 실제 장애나 유지보수 위험이 있을 때만 쓴다.
5. 출력은 위험도 높은 순서로 파일/라인 기준으로 짧게 쓴다.
/backend-review
이번 diff를 리뷰해줘.
상태 업데이트 로직의 트랜잭션, DB update 조건,
예외 처리, 누락된 JUnit 테스트만 봐줘.
# .claude/skills/java8-compat-review/SKILL.md
---
description: Review Java changes for Java 8 compatibility
allowed-tools: Read Grep Glob
---

Java 8 프로젝트에서 사용할 수 없는 문법이나 API를 찾는다.

확인할 항목:
1. List.of, Set.of, Map.of
2. var
3. Stream.toList
4. String.isBlank, Files.readString
5. java.net.http.HttpClient
6. sourceCompatibility 또는 targetCompatibility 변경

출력 형식:
- 파일/라인
- Java 8에서 문제가 되는 이유
- Java 8에서 사용할 대안
# .claude/skills/oracle11g-sql-review/SKILL.md
---
description: Review SQL for Oracle 11g compatibility
allowed-tools: Read Grep Glob
---

Oracle 11g에서 실행하기 어려운 SQL 패턴을 찾는다.

확인할 항목:
1. LIMIT, OFFSET, FETCH FIRST, FETCH NEXT
2. AUTO_INCREMENT, GENERATED AS IDENTITY
3. MySQL/PostgreSQL 함수: IFNULL, DATE_FORMAT, NOW
4. ROWNUM과 ORDER BY가 같은 query block에 있는 top-N 쿼리
5. ROWNUM > n 조건
6. MERGE에서 ON 조건 컬럼을 UPDATE하는 경우
7. sequence.NEXTVAL 없이 자동 증가 컬럼처럼 처리한 경우

대안 제안:
- top-N은 ORDER BY를 subquery 안에 두고 바깥에서 ROWNUM을 적용한다.
- pagination은 ROW_NUMBER() OVER (...)를 사용한다.
- 신규 ID는 sequence.NEXTVAL을 사용한다.
- IFNULL은 NVL, NOW()는 SYSDATE로 바꾼다.

출력 형식:
- 파일/라인
- Oracle 11g에서 문제가 되는 이유
- Oracle 11g에서 사용할 대안
/oracle11g-sql-review
이번 SQL 변경을 Oracle 11g 기준으로 리뷰해줘.
LIMIT, FETCH FIRST, OFFSET, identity column, ROWNUM 사용 순서,
MERGE 제약을 중심으로 봐줘.

이번 주 실천

최근에 세 번 이상 반복해서 설명한 요청을 하나 고르고, “언제 쓰는지, 어떤 순서로 보는지, 어떤 형식으로 답하는지”만 정리해봅니다.
