Week 4 of 8 반복 작업

Week 4

Skills로 반복 설명 줄이기

자주 시키는 일은 매번 길게 설명하지 않고, Skill로 작게 남겨둘 수 있습니다.

3줄 요약

  • 세 번 이상 반복해서 설명한 요청은 Skill 후보입니다.
  • Skill은 긴 프롬프트가 아니라 반복 작업의 순서와 답변 형식을 남겨두는 방법입니다.
  • 우리 팀에서는 리뷰, 테스트 보강, 장애 원인 분석부터 작게 시작하기 좋습니다.

이번 주 포인트

Skill은 잘 만든 프롬프트를 다시 꺼내 쓰기 쉽게 남겨두는 방법입니다. 일회성 설명을 줄이고, 비슷한 작업에서 참고할 수 있는 흐름을 만듭니다.

Claude Code를 몇 주 쓰다 보면 반복되는 요청이 보입니다. “커밋 메시지 써줘”, “diff 리뷰해줘”, “릴리즈 노트 정리해줘”, “테스트 빠진 부분 찾아줘.” 매번 원하는 형식과 우선순위를 설명하는 것은 낭비입니다.

Skill은 이런 반복 작업을 다시 설명하지 않게 도와주는 장치입니다. 좋은 Skill은 Claude에게 언제 이 작업 방식을 써보면 좋은지, 어떤 순서로 보면 편한지, 어떤 답변 형식이면 읽기 쉬운지를 알려줍니다.

1. Skill이 필요한 순간

Skill은 복잡한 지식보다 반복되는 작업 방식에 잘 맞습니다. 특히 팀마다 자주 보는 포인트가 다른 작업일수록 효과가 큽니다.

  • 커밋 메시지 작성
  • 코드 리뷰
  • 장애 원인 분석
  • 리팩토링 점검
  • 릴리즈 노트 정리
  • 테스트 보강 범위 찾기

2. 프롬프트와 Skill의 차이

프롬프트는 지금 한 번 필요한 지시입니다. Skill은 앞으로도 반복해서 꺼내 쓸 작업 방식입니다. 같은 요청을 세 번 이상 설명하고 있다면 Skill 후보로 봐도 좋습니다.

프롬프트

이번 diff 리뷰해줘.
버그랑 테스트 누락 위주로 보고,
스타일 얘기는 적게 해줘.

Skill

코드 리뷰 요청이면 항상:
1. 버그와 회귀 위험을 먼저 본다.
2. 파일/라인 기준으로 짧게 쓴다.
3. 스타일 의견은 실제 유지보수 위험이 있을 때만 쓴다.

3. 좋은 Skill의 구성

Skill은 길게 쓸수록 좋은 것이 아닙니다. Claude가 바로 따라 할 수 있는 짧은 흐름이 중요합니다.

  • 언제 이 Skill을 써보면 좋은지 적습니다.
  • 작업 순서를 짧게 남깁니다.
  • 읽기 편한 답변 형식을 적습니다.
  • 피하고 싶은 답변이나 작업 범위를 적습니다.
  • 필요한 참고 파일이나 명령이 있으면 연결합니다.

4. 작은 Skill이 낫다

하나의 거대한 Skill에 리뷰, 테스트, 릴리즈, 장애 분석을 모두 넣으면 Claude가 어느 포인트를 우선해야 하는지 흐려집니다. Skill은 역할이 작을수록 재사용하기 쉽고, 팀에서도 부담 없이 고치기 쉽습니다.

예를 들어 “PR 처리 Skill” 하나보다 “리뷰 Skill”, “릴리즈 노트 Skill”, “테스트 보강 Skill”로 나누는 편이 좋습니다. 작은 Skill은 수정하기도 쉽습니다. 리뷰할 때 보는 포인트가 바뀌면 리뷰 Skill만 조금 고치면 됩니다.

Java 백엔드 작업을 공통 사례로 보면 “상태 전환 리뷰 Skill”과 “테스트 보강 Skill”을 분리할 수 있습니다. 리뷰 Skill은 트랜잭션 범위, DB 업데이트 조건, 예외 처리 누락을 먼저 보게 하고, 테스트 보강 Skill은 성공, 실패, 경계 케이스를 기존 JUnit 스타일에 맞춰 찾게 합니다.

5. 우리 팀에서 시작하기 좋은 Skill 후보

처음부터 많은 Skill을 만들 필요는 없습니다. 이번 주 목표는 후보를 많이 만드는 것이 아니라, 팀에서 가장 자주 반복한 요청 하나를 골라 실제로 써보는 것입니다.

backend-review

  • PR이나 diff 리뷰 요청에 사용합니다.
  • 트랜잭션, DB 업데이트 조건, 예외 처리, 테스트 누락을 먼저 봅니다.
  • 출력은 파일/라인 기준으로 짧게 씁니다.

junit-test-planner

  • 버그 수정 전후에 필요한 JUnit 케이스를 찾습니다.
  • 성공, 실패, 경계, 재시도 케이스를 분리합니다.
  • 기존 테스트 스타일을 먼저 확인합니다.

issue-investigator

  • Jira 이슈 기반 원인 분석에 사용합니다.
  • 구현하지 않고 호출 흐름과 원인 후보만 정리합니다.
  • 확인할 로그, 테스트, 관련 파일을 남깁니다.

Java 8 호환성, Oracle 11g SQL, 릴리즈 노트처럼 자주 반복되는 요청이 있다면 다음 후보로 남겨둡니다. 하지만 첫 주에는 리뷰 Skill 하나만 실제 PR에 적용해보고, 쓸모없는 지적과 빠진 내용을 다듬는 편이 부담이 적습니다.

6. 팀 공통 Skill과 개인 Skill

같이 쓰는 Skill에는 자주 반복되는 작업 방식이 들어갑니다. 코드 리뷰 형식, 커밋 메시지 규칙, 릴리즈 노트처럼 여러 사람이 같이 보면 편한 작업입니다. 개인 Skill에는 개인의 요약 방식, 학습 방식, 선호하는 탐색 루틴을 둘 수 있습니다.

7. 팀에 적용하는 순서

처음부터 모든 Skill을 만들기보다 backend-review 하나를 실제 PR에 써보고 다듬는 것이 좋습니다. 아래 순서로 시작하면 부담이 적습니다. 

1. 저장소에 .claude/skills/backend-review/SKILL.md 초안을 만든다.
2. 작은 PR 하나를 골라 /backend-review로 리뷰를 요청해본다.
3. Claude가 낸 리뷰 중 쓸모없는 의견과 빠진 내용을 기록한다.
4. SKILL.md에서 답변 형식과 확인 순서를 조금 고친다.
5. 2~3번 써보고 괜찮으면 저장소에 남긴다.
6. 그 다음 junit-test-planner, java8-compat-review 같은 후보를 하나씩 써본다.

같이 쓰는 Skill은 한 번에 완성하려고 하지 않아도 됩니다. “우리 팀 리뷰에서 자주 놓치는 것”을 하나씩 반영하면 시간이 지날수록 실제 작업에 더 잘 맞아갑니다.

팀에서 가볍게 맞춰보면 좋은 것

“세 번 이상 반복해서 설명한 요청”은 Skill 후보로 기록합니다. Skill은 도구 설정이라기보다 반복 설명을 줄이는 작은 메모에 가깝습니다.

실전 Skill 예시

첫 공통 Skill은 아래처럼 작게 시작합니다.

# .claude/skills/backend-review/SKILL.md
---
description: Java 8 backend diff review for transaction, DB update, exception handling, and missing tests
allowed-tools: Read Grep Glob
---

Java backend diff를 리뷰할 때는 다음 순서로 본다.

1. 트랜잭션 범위와 DB 업데이트 조건을 먼저 확인한다.
2. 예외 처리, 재시도, 중복 처리 누락을 찾는다.
3. 기존 JUnit 테스트 스타일에 맞춰 빠진 케이스를 제안한다.
4. 스타일 의견은 실제 장애나 유지보수 위험이 있을 때만 쓴다.
5. 출력은 위험도 높은 순서로 파일/라인 기준으로 짧게 쓴다.
/backend-review
이번 diff를 리뷰해줘.
상태 업데이트 로직의 트랜잭션, DB update 조건,
예외 처리, 누락된 JUnit 테스트만 봐줘.

이번 주 실천

최근에 세 번 이상 반복해서 설명한 요청을 하나 고르고, “언제 쓰는지, 어떤 순서로 보는지, 어떤 형식으로 답하는지”만 정리해봅니다.

이번 주에 가볍게 남겨볼 것

같이 써볼 Skill 후보를 하나만 고르고, .claude/skills/backend-review/SKILL.md 같은 초안 파일로 남깁니다. 완성도가 낮아도 괜찮습니다. 실제 PR에 한 번 써본 뒤, 빠진 내용과 불필요한 의견을 고쳐보는 것이 이번 주 목표입니다.
