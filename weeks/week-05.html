Claude Code Engineering Roadmap

Week 5

MCP로 외부 시스템 연결하기

Claude Code가 코드만 보는 단계에서 벗어나 GitHub, 문서, 이슈, 사내 시스템까지 함께 볼 수 있으면 일의 단위가 달라집니다.

이번 주 핵심

MCP는 Claude에게 더 많은 도구를 주는 일이 아니라, 더 정확한 작업 맥락을 주는 일입니다.

버그 수정은 코드만 보고 끝나지 않는 경우가 많습니다. 이슈에 재현 절차가 있고, PR에 논의가 있고, 사내 문서에 설계 배경이 있고, 로그 시스템에 실제 실패 흔적이 있습니다. Claude가 코드만 읽으면 이 맥락을 사람이 계속 복사해서 붙여야 합니다.

MCP는 Claude Code가 외부 시스템을 조회할 수 있게 해주는 연결 방식입니다. 제대로 쓰면 “파일을 수정하는 도구”에서 “작업 맥락을 모아 판단하는 도구”로 바뀝니다.

1. MCP를 쓰는 이유

MCP의 가치는 연결 자체가 아니라 정보의 정확도에 있습니다. Claude가 PR diff, 이슈 설명, 문서, API 스펙을 직접 확인할 수 있으면 사람이 중간에서 맥락을 요약하다가 빠뜨리는 부분이 줄어듭니다.

특히 오래된 코드나 복잡한 서비스에서는 코드만 읽어서는 왜 이렇게 만들었는지 알기 어렵습니다. 외부 시스템은 그 이유를 담고 있는 경우가 많습니다.

2. 실무에서 유용한 연결

  • GitHub PR과 Issue
  • Jira, Linear 같은 이슈 트래커
  • 사내 문서와 ADR
  • 공식 API 문서
  • 로그와 모니터링 시스템

모든 시스템을 한 번에 붙일 필요는 없습니다. 팀이 실제로 자주 참고하는 신뢰 가능한 출처부터 연결하는 것이 좋습니다. 연결이 많아질수록 Claude가 어떤 정보를 우선해야 하는지도 함께 정해야 합니다.

3. MCP가 잘 맞는 작업

MCP는 “코드 변경 전에 확인해야 하는 외부 맥락”이 있는 작업에서 특히 유용합니다.

잘 맞는 작업

  • PR 리뷰
  • 이슈 기반 버그 수정
  • 릴리즈 영향 범위 파악
  • 문서와 코드 불일치 찾기

덜 필요한 작업

  • 작은 오타 수정
  • 단일 파일 정리
  • 명확한 테스트 추가
  • 이미 맥락이 대화에 모두 있는 작업

4. 권한과 보안

MCP는 편리하지만 권한을 신중하게 다뤄야 합니다. 처음에는 읽기 권한부터 시작하는 편이 좋습니다. 쓰기 작업은 댓글 작성, 이슈 수정, PR 업데이트처럼 실제 협업 기록을 바꾸기 때문에 명시적인 승인 흐름을 두는 것이 안전합니다.

특히 사내 문서, 고객 로그, 운영 시스템은 접근 범위가 넓어질수록 리스크도 커집니다. Claude가 볼 수 있는 정보와 할 수 있는 작업을 분리해서 생각해야 합니다.

5. 실전 요청 예시

이 이슈를 기준으로 원인을 조사해줘.
먼저 이슈 설명과 관련 PR 논의를 확인하고,
코드에서 관련 흐름을 찾은 뒤,
수정 계획과 필요한 테스트를 정리해줘.
쓰기 작업은 내가 승인하기 전에는 하지 마.

좋은 요청은 MCP를 “많이 써라”가 아니라 “어떤 출처를 어떤 순서로 확인하라”에 가깝습니다. 신뢰할 수 있는 소스를 지정하면 Claude의 판단도 안정됩니다.

팀에서 합의하면 좋은 기준

MCP는 읽기부터 시작하고, 쓰기 권한은 작업 종류별로 별도 승인 기준을 둡니다. 연결 개수보다 신뢰할 수 있는 출처의 품질이 중요합니다.
