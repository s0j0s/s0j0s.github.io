Week 5

MCP로 외부 시스템 연결하기

Claude Code가 코드만 보는 단계에서 벗어나 GitHub, Jira, Confluence까지 함께 볼 수 있으면 일의 단위가 달라집니다.

3줄 요약

  • MCP는 Claude Code에게 더 많은 기능을 주는 것이 아니라 더 정확한 작업 맥락을 주는 방법입니다.
  • GitHub, Jira, Confluence는 읽기 권한부터 연결하고, 쓰기 작업은 한 번 확인하고 맡깁니다.
  • 실전에서는 Jira 이슈, Confluence 설계 문서, GitHub PR 논의를 한 흐름으로 확인하게 만듭니다.

이번 주 포인트

MCP는 Claude에게 더 많은 도구를 주는 일이 아니라, 더 정확한 작업 맥락을 주는 일입니다. 권한은 읽기부터 시작하고, 쓰기는 한 번 확인하고 맡기는 편이 좋습니다.

버그 수정은 코드만 보고 끝나지 않는 경우가 많습니다. 이슈에 재현 절차가 있고, PR에 논의가 있고, Confluence에 설계 배경이 있고, Jira에 업무 히스토리가 있습니다. Claude가 코드만 읽으면 이 맥락을 사람이 계속 복사해서 붙여야 합니다.

MCP는 Claude Code가 외부 시스템을 조회할 수 있게 해주는 연결 방식입니다. 제대로 쓰면 “파일을 수정하는 도구”에서 “작업 맥락을 모아 판단하는 도구”로 바뀝니다.

1. MCP를 쓰는 이유

MCP의 가치는 연결 자체가 아니라 정보의 정확도에 있습니다. Claude가 PR diff, 이슈 설명, 문서, API 스펙을 직접 확인할 수 있으면 사람이 중간에서 맥락을 요약하다가 빠뜨리는 부분이 줄어듭니다.

특히 오래된 코드나 복잡한 서비스에서는 코드만 읽어서는 왜 이렇게 만들었는지 알기 어렵습니다. 외부 시스템은 그 이유를 담고 있는 경우가 많습니다. 상태 업데이트 버그를 조사할 때는 Jira의 재현 절차, Confluence의 상태 전환 설계, GitHub의 관련 PR 논의를 MCP로 먼저 확인하면 코드 변경 전에 판단 근거가 선명해집니다.

2. 실무에서 유용한 연결

  • GitHub PR, diff, review comment
  • Jira 이슈, 재현 절차, 상태, 담당자
  • Confluence 설계 문서, 운영 가이드, 의사결정 기록
  • 사내 API 문서와 배포 체크리스트
  • 로그와 모니터링 시스템

모든 시스템을 한 번에 붙일 필요는 없습니다. 팀이 실제로 자주 참고하는 신뢰 가능한 출처부터 연결하는 것이 좋습니다. 연결이 많아질수록 Claude가 어떤 정보를 먼저 보면 좋을지도 가볍게 적어두면 편합니다.

3. 먼저 얘기해두면 좋은 권한 범위

MCP를 팀에 적용할 때는 연결보다 권한 범위를 먼저 얘기해두는 편이 좋습니다. 읽기 작업은 조사 품질을 높이는 데 쓰고, 협업 기록을 바꾸는 쓰기 작업은 한 번 확인한 뒤 맡깁니다.

읽기 허용

  • Jira 이슈 설명과 재현 조건 조회
  • Confluence 설계 문서 조회
  • GitHub PR, diff, review comment 조회

확인 후 쓰기

  • Jira 이슈에 조사 결과 댓글 작성
  • PR 설명 업데이트
  • 리뷰 코멘트 작성

별도 논의

  • 이슈 상태 변경
  • 브랜치 삭제나 force push
  • 운영 로그, 고객정보, 배포 관련 작업

4. GitHub, Jira, Confluence MCP 연동 단계

실제 설정 방법은 회사 계정, 보안 정책, 사용하는 MCP 서버에 따라 달라질 수 있습니다. 그래도 팀에서 가볍게 맞춰두면 좋은 순서는 비슷합니다. 

1. 연결할 시스템을 고른다.
   GitHub PR, Jira 이슈, Confluence 문서처럼 자주 보는 출처부터 시작한다.

2. 권한을 읽기 전용으로 시작한다.
   PR 조회, 이슈 조회, 문서 조회까지만 허용하고 쓰기 작업은 막아둔다.

3. 토큰과 접근 범위를 분리한다.
   개인 토큰을 넓게 쓰지 말고, 필요한 저장소와 프로젝트 범위만 허용한다.

4. Claude에게 읽는 순서를 정해준다.
   Jira -> Confluence -> GitHub -> 코드 순서처럼 읽는 흐름을 정해본다.

5. 쓰기 작업은 한 번 더 확인한다.
   댓글 작성, 이슈 상태 변경, PR 수정은 사람이 확인한 뒤에만 실행한다.

처음 연결할 때는 “잘 연결되었는지”보다 “무엇을 읽을 수 있고 무엇을 할 수 없는지”를 팀원이 이해하는 것이 더 중요합니다. 읽기 권한만으로도 조사 품질은 충분히 좋아집니다.

5. 실제 연결 예시

아래 예시는 개인 토큰을 코드에 넣지 않고 환경변수로 넘기는 방식입니다. 토큰은 필요한 저장소와 프로젝트만 볼 수 있도록 범위를 좁혀서 발급합니다. Atlassian 주소는 예시이므로, 실제로는 회사에서 운영하거나 확인한 MCP 서버 주소로 바꿔주세요. 

# GitHub MCP 연결
export GITHUB_PAT=your_github_fine_grained_token

claude mcp add --transport http github https://api.githubcopilot.com/mcp/ \
  --header "Authorization: Bearer $GITHUB_PAT"

# Jira, Confluence MCP 연결
export ATLASSIAN_TOKEN=your_atlassian_token

claude mcp add --transport http atlassian https://company-atlassian-mcp.example.com/mcp \
  --header "Authorization: Bearer $ATLASSIAN_TOKEN"

# 연결 확인
claude mcp list

Claude Code 안에서는 /mcp로 연결 상태와 인증 상태를 확인합니다. 팀 공통으로 배포할 MCP는 project scope로 관리할 수 있지만, 토큰 값은 절대 저장소에 넣지 않습니다. 

Jira MES-1234를 읽고 재현 조건과 기대 동작을 정리해줘.
그 다음 Confluence의 상태 전환 설계 문서를 확인하고,
GitHub에서 관련 PR과 최근 변경을 찾아줘.
아직 코드는 수정하지 말고 원인 후보와 확인할 테스트만 알려줘.

6. MCP가 잘 맞는 작업

MCP는 “코드 변경 전에 확인해야 하는 외부 맥락”이 있는 작업에서 특히 유용합니다.

잘 맞는 작업

  • PR 리뷰
  • 이슈 기반 버그 수정
  • 릴리즈 영향 범위 파악
  • 문서와 코드 불일치 찾기

덜 필요한 작업

  • 작은 오타 수정
  • 단일 파일 정리
  • 명확한 테스트 추가
  • 이미 맥락이 대화에 모두 있는 작업

7. GitHub, Jira, Confluence를 함께 쓰는 흐름

연결을 많이 하는 것보다, 어떤 순서로 읽게 할지가 중요합니다. 상태 업데이트 버그를 예로 들면 다음 순서가 무난합니다. 

1. Jira에서 이슈 설명, 재현 조건, 기대 동작을 확인한다.
2. Confluence에서 상태 전환 규칙과 예외 케이스를 확인한다.
3. GitHub에서 관련 PR, 최근 변경, review comment를 확인한다.
4. 코드에서 호출 흐름과 테스트를 찾는다.
5. 수정 계획과 검증 방법을 제안한다.

8. 권한과 보안

MCP는 편리하지만 권한은 신중하게 다루는 편이 좋습니다. 처음에는 읽기 권한부터 시작하면 부담이 적습니다. 쓰기 작업은 댓글 작성, 이슈 수정, PR 업데이트처럼 실제 협업 기록을 바꾸기 때문에 실행 전에 한 번 확인하는 흐름이 있으면 안전합니다.

읽기 권한은 조사 품질을 높이는 데 쓰고, 쓰기 권한은 협업 기록을 바꾸는 행동으로 따로 봐야 합니다. 예를 들어 “이슈에 조사 결과를 댓글로 남기기”, “PR 설명 업데이트하기”, “이슈 상태 바꾸기”는 모두 팀의 기록을 바꾸므로 진행 전에 한 번 확인하는 편이 좋습니다.

특히 사내 문서, 고객 로그, 운영 시스템은 접근 범위가 넓어질수록 리스크도 커집니다. Claude가 볼 수 있는 정보와 할 수 있는 작업을 분리해서 생각하는 편이 좋습니다.

9. 실전 요청 예시

그대로 써봐도 되는 요청

Jira 이슈를 기준으로 작업 맥락을 모을 때 사용할 수 있습니다.

Jira 이슈 MES-1234를 기준으로 원인을 조사해줘.
먼저 Jira의 재현 조건과 기대 동작을 읽고,
Confluence에서 관련 상태 전환 규칙을 확인해줘.
그 다음 GitHub에서 관련 PR과 최근 변경을 찾고,
코드의 호출 흐름, 수정 계획, 필요한 테스트를 정리해줘.
댓글 작성이나 이슈 상태 변경 같은 쓰기 작업은 내가 한 번 확인하기 전에는 하지 마.

좋은 요청은 MCP를 “많이 써라”가 아니라 “어떤 출처를 어떤 순서로 확인하라”에 가깝습니다. 신뢰할 수 있는 소스를 지정하면 Claude의 판단도 안정됩니다.

팀에서 가볍게 맞춰보면 좋은 것

MCP는 읽기부터 시작하고, 쓰기 작업은 작업 종류별로 한 번 더 확인합니다. 연결 개수보다 신뢰할 수 있는 출처의 품질이 중요합니다.

Jira 이슈 하나를 골라 Claude Code에게 Jira, Confluence, GitHub 순서로 맥락을 확인하게 하고, 코드 수정 전 조사 결과만 받아봅니다.

이번 주에 가볍게 남겨볼 것

MCP 연결 목록보다 먼저 권한 범위를 메모해봅니다. “읽기부터 써보기”, “확인 후 쓰기”, “별도 논의” 세 칸으로 나누어 GitHub, Jira, Confluence에서 Claude Code에게 맡겨볼 일을 적어둡니다.
