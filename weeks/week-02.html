Week 2 of 8 작업 루프

Week 2

바로 구현시키지 않는 Claude Code 작업 루프

같은 버그도 어떻게 맡기느냐에 따라 결과물, 테스트 근거, 리뷰 부담이 달라집니다.

3줄 요약

  • “바로 고쳐줘”는 빠르지만 조사 근거와 수정 범위가 흐려지기 쉽습니다.
  • 좋은 작업 루프는 조사, 계획, 구현, 테스트, diff 리뷰로 닫힙니다.
  • 이번 주에는 버그 수정을 맡길 때 루프 전체를 한 번에 요청해보겠습니다.

이번 주 핵심

Week 1에서 “먼저 조사해줘”를 붙여보자고 했습니다. Week 2에서는 그 다음 단계로, 조사부터 diff 리뷰까지 이어지는 기본 작업 루프를 하나의 요청 패턴으로 정리합니다.

Claude Code는 빠르게 코드를 읽고 수정할 수 있습니다. 그래서 급한 버그를 만나면 바로 “고쳐줘”라고 말하고 싶어집니다. 하지만 팀에서 리뷰해야 하는 변경이라면 결과물만큼 중요한 것이 있습니다. 왜 그 파일을 바꿨는지, 어떤 테스트로 확인했는지, 남은 위험은 무엇인지입니다.

이번 글에서는 같은 버그를 두 가지 방식으로 맡겨봅니다. 하나는 바로 구현 요청이고, 다른 하나는 조사와 계획을 먼저 받는 요청입니다. 차이는 프롬프트 길이가 아니라, 리뷰 가능한 근거를 남기느냐에 있습니다.

이번 글의 예시 버그

특정 조건에서 작업 상태 업데이트 API가 성공처럼 응답하지만, DB에는 기대한 상태가 반영되지 않는 상황을 기준으로 보겠습니다. Java 백엔드에서 자주 만나는 상태 전환, DB 업데이트 조건, 트랜잭션 범위가 얽힌 버그입니다.

1. 같은 버그, 두 가지 요청

먼저 가장 흔한 요청입니다.

부족한 요청

작업 상태 업데이트 버그 고쳐줘.

더 나은 요청

바로 구현하지 말고 먼저 원인을 조사해줘.
관련 파일, 현재 동작, 재현 조건을 정리하고
가장 작은 수정 계획과 테스트 방법을 제안해줘.

두 번째 요청은 Claude에게 먼저 근거를 만들게 합니다. 근거가 있으면 사람이 방향을 확인할 수 있고, 구현이 잘못된 곳으로 달리는 것을 줄일 수 있습니다.

2. 결과는 이렇게 달라집니다

같은 작업이라도 요청 방식이 달라지면 리뷰할 때 확인할 수 있는 정보가 달라집니다.

바로 구현 요청

  • 수정 속도는 빠를 수 있습니다.
  • 왜 그 파일을 바꿨는지 다시 물어봐야 할 수 있습니다.
  • 테스트 근거가 부족하면 리뷰 부담이 커집니다.

조사 후 구현 요청

  • 관련 파일과 현재 동작을 먼저 확인합니다.
  • 수정 범위와 테스트 방법이 먼저 보입니다.
  • diff 리뷰에서 남은 리스크를 확인하기 쉽습니다.

바로 구현 요청 후 리뷰에서 남는 질문

왜 이 파일을 바꿨는가, 다른 상태값에도 영향이 있는가, 어떤 테스트로 확인했는가를 다시 확인해야 합니다. 처음부터 이 근거를 만들게 하면 리뷰가 훨씬 짧아집니다.

3. 기본 작업 루프

목표 정의
-> 코드 조사
-> 변경 계획
-> 구현
-> 테스트
-> diff 리뷰
-> 수정

목표 정의에서는 원하는 결과와 제약을 말합니다. 코드 조사에서는 Claude가 관련 파일, 호출 흐름, 기존 테스트를 찾습니다. 변경 계획에서는 수정 범위와 검증 방법을 정리합니다. 구현은 그 다음입니다.

테스트 후에도 끝이 아닙니다. Claude에게 diff 기준으로 리스크를 다시 보게 해야 합니다. “내가 무엇을 바꿨고, 어떤 동작이 영향을 받을 수 있으며, 어떤 테스트로 확인했는가”를 말하게 하면 리뷰가 훨씬 쉬워집니다.

4. 단계별로 이렇게 맡겨볼 수 있습니다

매번 긴 설명을 새로 쓸 필요는 없습니다. 단계별로 자주 쓰는 문장을 정해두면 됩니다. 

조사:
관련 파일과 호출 흐름을 먼저 정리해줘.

계획:
가장 작은 수정안을 제안하고, 바뀔 수 있는 동작을 알려줘.

구현:
계획한 범위 안에서만 수정하고, 불필요한 리팩토링은 하지 말아줘.

테스트:
어떤 테스트로 확인했는지와 실패한다면 남은 원인을 정리해줘.

리뷰:
diff 기준으로 회귀 위험, 누락된 테스트, 애매한 가정을 먼저 봐줘.

5. 언제는 바로 맡겨도 될까

모든 작업에 긴 루프가 필요한 것은 아닙니다. 오타 수정, 주석 정리, 로그 문구 변경처럼 영향 범위가 작은 작업은 바로 맡겨도 괜찮습니다. 반대로 다음 조건 중 하나라도 있으면 먼저 조사와 계획을 받는 편이 좋습니다.

  • 수정 파일이 여러 영역에 걸칠 가능성이 있다.
  • 기존 설계 의도를 알아야 한다.
  • 트랜잭션 범위, DB 업데이트 조건, 상태 전환처럼 리스크가 크다.
  • 배치 처리, 외부 시스템 연동, 예외 처리, 재시도 로직에 영향이 있다.
  • 테스트가 실패했는데 원인이 명확하지 않다.
  • 리팩토링처럼 정답보다 판단이 중요한 작업이다.

복사해서 써볼 요청

버그 수정이나 테스트 실패를 맡길 때 그대로 붙여볼 수 있습니다.

이 작업은 바로 구현하지 말고 다음 순서로 진행해줘.
1. 관련 파일과 현재 동작을 조사해줘.
2. 원인 후보와 가장 작은 수정 계획을 제안해줘.
3. 계획한 범위 안에서 구현해줘.
4. 테스트 또는 확인 방법을 실행하고 결과를 정리해줘.
5. 마지막에 diff 기준으로 리스크를 리뷰해줘.

이번 주 실천

버그 수정 요청을 할 때 구현 전 조사, 수정 계획, 테스트, diff 리뷰까지 한 번에 요청해봅니다.
