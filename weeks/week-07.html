Week 7

Hooks로 잊기 쉬운 확인 알려주기

약 6분 · 실무 활용 팁

팀에서 Claude Code를 안정적으로 쓰려면 좋은 요청뿐 아니라 잊기 쉬운 확인을 알려주는 작은 장치가 있으면 편합니다.

3줄 요약

  • Hooks는 Claude Code 작업 중 반복 확인을 자동으로 알려주는 리마인더입니다.
  • 처음부터 막으려 하지 말고 알림부터 시작하면 팀원이 부담 없이 받아들일 수 있습니다.
  • Java 백엔드 팀이라면 관련 JUnit 테스트, 정적 검사, secret 검사부터 연결하기 좋습니다.

이번 주 포인트

Hooks는 Claude를 믿지 못해서 거는 장치가 아니라, 자주 잊는 확인을 작업 흐름 안에서 알려주는 리마인더입니다. 도입은 자동 실행보다 가벼운 알림부터 시작하는 편이 편합니다.

Claude에게 매번 “테스트 돌려줘”, “포맷 확인해줘”, “secret 들어갔는지 봐줘”라고 말하는 것은 오래 가지 않습니다. 사람도 반복 확인을 가끔 빠뜨리는데, Claude에게 매번 대화로만 지시하는 방식은 더 쉽게 흔들립니다.

Hooks는 특정 시점에 자동으로 실행되는 작은 알림입니다. 변경 후 테스트를 떠올리게 하거나, 위험한 명령을 한 번 더 확인하거나, 특정 파일이 수정될 때 메시지를 띄우는 식으로 반복 확인을 작업 흐름 안에 넣을 수 있습니다.

1. 왜 필요한가

반복 확인을 기억에만 맡기면 가끔 빠질 수 있습니다. 특히 Claude Code를 쓰면 변경 속도가 빨라지기 때문에 검증 루프가 따라오지 못할 수 있습니다. Hooks는 이 간격을 줄입니다.

중요한 점은 Hooks가 모든 판단을 자동화하는 것이 아니라는 점입니다. 사람이 항상 기억하지 않아도 되는 기본 확인을 자연스럽게 떠올리게 하는 데 목적이 있습니다.

2. 실무 Hook 예시

  • 변경 후 관련 JUnit 테스트 실행
  • 정적 검사와 코드 스타일 확인
  • secret 검사
  • 위험 명령 실행 전 확인
  • 트랜잭션, 배치, 외부 연동 파일 수정 시 알림
  • 상태 업데이트 관련 파일 수정 시 관련 테스트 실행 안내
  • PR 전 체크리스트 출력

예를 들어 상태 업데이트 서비스나 외부 연동 어댑터가 바뀌면 관련 JUnit 테스트나 ./gradlew test를 떠올리게 하는 Hook부터 시작할 수 있습니다. 팀이 안정성을 확인한 뒤에는 같은 Hook을 자동 실행으로 올리고, 민감정보 유출이나 위험 명령처럼 명확한 리스크만 제한적으로 실행을 멈춥니다.

3. Hook을 만드는 순서

처음 Hook을 만들 때는 자동 실행보다 알림 메시지부터 만드는 편이 좋습니다. 팀원이 실제 흐름에서 불편하지 않은지 확인한 뒤 자동 실행이나 제한적인 실행 중단을 검토합니다. 

1. 자주 놓치는 검증을 하나 고른다.
   예: 상태 업데이트 서비스 변경 후 관련 JUnit 테스트 확인

2. 감지할 파일 범위를 골라본다.
   예: *StatusService.java, *StateRepository.java, 외부 연동 adapter

3. 처음에는 알림만 출력한다.
   예: "상태 업데이트 로직이 변경되었습니다. 관련 JUnit 테스트를 확인하세요."

4. 팀에서 괜찮다고 느끼면 자동 실행으로 올린다.
   예: ./gradlew test --tests "*Status*"

5. 정말 위험한 작업만 실행을 멈춘다.
   예: secret 노출, 확인 없는 force push, 운영 설정 파일 삭제

4. 실제 Hook 파일 예시

처음에는 파일 경로를 파싱하지 않는 단순 Hook부터 시작해도 충분합니다. 아래 예시는 Claude가 파일을 수정한 뒤 테스트, Java 8, Oracle 11g 확인을 잊지 않도록 알림만 출력합니다. 

# .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PROJECT_DIR}/.claude/hooks/test-reminder.sh"
          }
        ]
      }
    ]
  }
}
# .claude/hooks/test-reminder.sh
#!/usr/bin/env bash

echo "Files were changed. Before finishing, ask Claude to check related JUnit tests." >&2
echo "If Java code changed, check Java 8 compatibility." >&2
echo "If SQL, MyBatis XML, or JDBC query changed, check Oracle 11g compatibility." >&2

exit 0
chmod +x .claude/hooks/test-reminder.sh

이 버전은 별도 도구 설치가 필요 없습니다. 팀원이 알림을 부담 없이 받아들이는지 먼저 확인한 뒤, 필요할 때 파일별로 다른 알림을 띄우는 정교한 Hook으로 확장합니다.

이 Hook이 팀 흐름에 맞는지 확인한 뒤 더 정교한 검사를 붙입니다. 예를 들어 Java 파일이 바뀌면 Java 8 호환성을 확인하라고 알려주거나, SQL, MyBatis XML, JDBC query 문자열이 바뀌면 Oracle 11g 호환성을 확인하라고 안내할 수 있습니다. 처음부터 모든 검사를 자동 실행하거나 막으려 하지 않으면 부담이 적습니다.

5. 추천 Hook Set

test-reminder

  • 서비스, repository, adapter 변경 시 관련 JUnit 테스트를 안내합니다.
  • 처음에는 알림만 출력합니다.
  • 안정화되면 제한된 테스트 자동 실행으로 올립니다.

secret-guard

  • 토큰, 비밀번호, 인증키 패턴을 검사합니다.
  • 명확한 민감정보 패턴은 실행 중단 후보입니다.
  • 오탐이 많으면 알림 단계로 낮춥니다.

risk-command-guard

  • 삭제, 강제 push, 권한 변경 같은 위험 명령을 확인합니다.
  • Claude가 실행하기 전 사람이 한 번 확인하게 합니다.
  • 팀에서 허용한 명령은 예외 목록으로 관리합니다.

pr-checklist

  • 작업 종료 전에 테스트, diff 리스크, 문서 변경 여부를 출력합니다.
  • 리뷰 요청 전에 빠진 근거를 확인하게 도와줍니다.
  • 막기보다 체크리스트 출력에 적합합니다.

oracle11g-sql-guard

  • .sql, .xml, .java 변경 시 SQL 문법을 확인합니다.
  • Oracle 12c+, MySQL, PostgreSQL 문법이 섞이면 알려줍니다.
  • 처음에는 알림만 두고, 오탐이 적은 패턴만 실행 중단 후보로 둡니다.

6. 팀에 적용하는 순서

Hook은 적용 순간부터 개발 흐름에 영향을 줍니다. 그래서 첫 주에는 알림만 켜고, 팀원이 실제로 불편함 없이 받아들이는지 확인하는 편이 좋습니다. 

1. .claude/hooks/test-reminder.sh를 단순 알림 버전으로 만든다.
2. .claude/settings.json에 test-reminder Hook만 연결한다.
3. 작은 Java 파일 하나를 Claude Code로 수정해보고 알림이 뜨는지 확인한다.
4. 팀원이 알림을 부담 없이 받아들이면 저장소에 남긴다.
5. 1~2주 사용한 뒤 자동 실행이나 실행 중단으로 올릴 항목만 따로 고른다.
6. Java 8, Oracle 11g, secret 검사는 오탐이 적은 항목부터 하나씩 추가한다.

처음부터 ./gradlew test를 매번 자동 실행하면 느릴 수 있습니다. 먼저 “관련 테스트를 확인하라”는 알림을 써보고, 이후 특정 파일 패턴에만 제한적으로 테스트 자동 실행을 붙이는 것이 현실적입니다.

7. 자동화할 것과 사람이 판단할 것

자동화하기 좋은 것은 결과가 명확한 작업입니다. 테스트 통과 여부, 코드 스타일 위반 여부, secret 패턴 검출, 위험 명령 확인 같은 것은 Hook에 잘 맞습니다.

반대로 제품 의도, 설계 방향, 리스크 수용 여부는 사람이 판단하는 편이 좋습니다. Hook은 사람의 판단을 대체하기보다 판단 전에 놓치면 안 되는 기본 신호를 제공합니다.

Hook에 잘 맞음

  • 테스트 명령 실행
  • 코드 스타일 검사
  • 민감정보 패턴 검사
  • 주의할 파일 변경 알림

사람이 판단

  • 변경 방향 확인
  • 배포 리스크 수용
  • 설계 트레이드오프
  • 제품 요구사항 해석

8. 너무 강한 Hook의 문제

모든 것을 막는 방식으로 시작하면 개발 흐름이 막힙니다. 테스트가 느리거나 flaky한데 매번 강제하면 Claude Code 사용 자체가 답답해질 수 있습니다. 처음부터 완벽한 방어선을 만들려고 하기보다 팀이 실제로 놓치는 문제부터 잡는 편이 좋습니다.

그대로 써봐도 되는 요청

팀에서 첫 Hook 후보를 정할 때 사용할 수 있습니다.

최근 변경에서 자주 수정되는 Java 백엔드 파일을 기준으로
Hook으로 알려주면 좋을 항목을 제안해줘.
처음부터 막으려 하지 말고 알림으로 시작할 항목,
자동 실행해도 괜찮은 테스트,
사람 확인이 필요한 위험 작업을 나눠서 정리해줘.

이번 주 실천

실행을 멈추는 Hook을 만들기 전에 “상태 업데이트 관련 파일이 바뀌면 관련 테스트를 확인하라”는 test-reminder 알림 하나만 먼저 써봅니다. 팀이 1~2주 사용한 뒤 자동 실행이나 실행 중단으로 올릴 항목을 따로 고릅니다.
