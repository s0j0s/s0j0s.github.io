Week 6 · 독립 작업

SubAgent로 큰 작업
나누기

긴 조사와 분석을 별도 맥락에 맡기고
메인 대화에는 필요한 결과만 가져옵니다.

키보드 방향키 또는 아래 버튼으로 넘겨보세요.

한 대화에 모든 일을 담으면

조사하다 구현하고,
마지막에는 리뷰까지 섞입니다.

중간 정보가 길어질수록 지금 무엇을 판단해야 하는지 흐려집니다.

SubAgent가 맡는 일

별도의 맥락에서 일하고
최종 결과를 돌려줍니다.

  1. 1독립된 역할조사, 로그 분석, 의존성 점검
  2. 2제한된 도구역할에 필요한 읽기·검색만 허용
  3. 3정해진 결과관련 파일, 원인 후보, 검증 방법

메인 대화에는 SubAgent의 최종 답변과 기본 실행 정보만 돌아옵니다.

Skill과 SubAgent

과정을 함께 볼지,
결과만 받을지 고릅니다.

메인 대화에서 진행 Skill

각 단계를 보면서 질문하고 방향을 조정

별도 맥락에서 진행 SubAgent

긴 중간 과정은 분리하고 최종 결과만 전달

언제 나눌까

맥락 부담이 큰 옆 작업에
SubAgent가 잘 맞습니다.

써볼 만한 작업 원인 후보가 많은 버그
긴 로그 분석
독립적인 diff 리뷰

중간 결과를 메인 대화에서 다시 쓰지 않을 때

기본 루프로 충분 오타 수정
단일 테스트 추가
명확한 조건문 수정

역할을 나누는 시간이 수정 시간보다 길 때

우리 팀의 첫 역할

구현보다 먼저
읽기 전용 조사부터 맡깁니다.

  1. 1Issue Investigator호출 흐름, 관련 파일, 원인 후보
  2. 2Small Fix Builder사람이 고른 작은 수정과 테스트
  3. 3Diff Reviewer회귀 위험, 테스트 누락, 과한 변경

처음에는 파일을 수정하지 않는 Investigator 하나만 공유해도 충분합니다.

역할을 나누는 흐름

결과를 확인한 뒤
다음 역할로 넘깁니다.

  1. Step 1Investigator가 원인 후보 조사
  2. Step 2사람이 수정 방향과 범위 선택
  3. Step 3Builder가 작은 수정과 테스트 수행
  4. Step 4Reviewer가 실제 diff 검토

앞 결과에 의존하는 작업은 한꺼번에 병렬로 보내지 않습니다.

SubAgent 파일

할 일뿐 아니라
하지 않을 일도 적습니다.

프로젝트 역할 정의 .claude/agents/
issue-investigator.md

이름, 설명, 모델, 사용할 도구

결과 계약 범위 · 제외 · 출력

수정 금지, 원인 후보 2개, 테스트 계획

실전 요청

역할과 반환할 결과를
짧게 지정합니다.

issue-investigator agent로 MES-1234 원인을 조사해줘.

파일은 수정하지 말고 다음 결과만 정리해줘.
- 관련 코드 위치와 호출 흐름
- 가능성 높은 원인 2개
- 확인해야 할 JUnit 테스트

남용하면 생기는 일

역할이 많아질수록
통합과 조율 비용도 커집니다.

  • 같은 파일을 여러 역할이 중복해서 읽음
  • 조사 결과와 구현 범위가 서로 어긋남
  • 작은 수정에 실제 작업보다 설명이 더 길어짐
  • 최종 판단의 책임이 흐려짐

어떤 결과를 채택하고 무엇을 수정할지는 사람이 마지막에 고릅니다.

이번 주 실천

최근 이슈 하나를
Investigator에게만 맡겨봅니다.

결과가 충분히 명확할 때만 작은 구현으로 넘어갑니다.

한 문장으로 정리

분리할 가치가 있는
옆 작업에 씁니다.
